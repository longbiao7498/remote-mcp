# remote-mcp v0.2.0 设计规范：非持久 Bash + 可配置 cwd

**日期**：2026-05-27  
**状态**：待审核 → 待实施  
**前身**：增量于 [`2026-05-26-remote-mcp-design.md`](./2026-05-26-remote-mcp-design.md)（v2, 已实施为 v0.1.x）；本规范覆盖 v0.2.0 的架构转向，其他设计（连接生命周期、SFTP 工具、9 个非 Bash 工具）继续以 v2 spec 为准。

---

## 1. 概述与目标

v0.2.0 包含两项主要变更，二者都服务于同一个目标——**对齐 Claude Code 原生使用体验**：

1. **非持久 Bash**——Bash 工具从持久 bash session 模型改为每次调用独立 channel 的非持久模型（§3-§5）
2. **可配置 cwd**——MCP 注册时通过 `--cwd` 指定远程工作目录，所有工具的相对路径以此为锚；工具输出统一后缀 `[host=X cwd=Y]`（§6）

**核心目标**（沿用项目原始目标 §1）：让 Claude Code 通过 MCP 操控远程 Linux 服务器，**使用体验对齐到不会因 schema/输出格式/行为差异导致 agent 用错工具**。

v0.1.x 在这条目标上有一个长期未发现的盲点：**Bash 工具的"shell 持久性"与 Claude Code 原生工具行为不一致**——原生每次调用是新 shell，我们的是持久 session。这个盲点带来：

1. 一类系统性 bug（srun 等 stdin-watcher 程序在持久 PTY 上必然挂死）
2. agent 心智负担（"我上一调用 export 的变量这次还在吗？cd 了吗？"）
3. ~350 LOC 的支撑机制（sentinel 协议、PTY 分配、reader thread、setsid 包装等）

v0.2.0 通过架构转向同时解决这三件事。

## 3. 动机（目标导向重述）

之前的设计推理走过这样一条路：

```
v1（原始决策）：
  "持久 bash 让 cwd/env 跨调用持久，agent UX 更好" → 选了持久

v0.1.0-v0.1.2（逐渐暴露的代价）：
  - sentinel 协议（必要，因为持久 stdout 无命令边界）
  - PTY 分配（必要，因为没 PTY 时 \x03 不发 SIGINT）
  - reader thread（必要，避免缓冲区死锁）
  - setsid 包装后台命令（必要，否则 kill PGID 会误杀 BashSession）
  - </dev/null 包装（必要，否则 srun 类 stdin-watcher 挂死）
  - timeout 时返回 partial output（必要，否则信息丢失）
  
v0.2.0 触发点（agent 反馈 + 实测）：
  - 反馈：srun + timeout-丢-stdout 的复合诊断
  - 实测：Claude Code 原生 Bash 每次重置 cwd/env，跟我们持久行为不同
  - 反思：v1 §3.6 把"持久 bash"当作设计基石，但其实它服务的是
         "人类视角的便利"而非"agent 视角的对齐"；目标里说的是
         后者，所以原决策与目标其实不一致
```

→ v0.2.0 的转向不是修一个 bug，是**修一个 v1 时的设计前提错误**。

## 4. 架构转向

### 4.1 SSH 三层中谁动谁不动

```
┌──────────────────────────────────────────────────────────┐
│  TCP 连接 — 不变（一辈子 1 个）                            │
│  SSH Transport — 不变（一辈子 1 个；compress, keepalive）  │
│  ├─ Channel: SFTP — 不变（懒初始化，复用）                  │
│  ├─ Channel: persistent bash — 删除                       │
│  └─ Channel: exec ×N — 不变 + 范围扩大                     │
│      │   (Glob/Grep/MultiRead/Read sed-slice)             │
│      └─ 新成员：Bash 每次调用 ← 这是 v0.2.0 的全部架构改动  │
└──────────────────────────────────────────────────────────┘
```

**Bash 从"占一个常驻 channel"变成"每次开一个一次性 channel，跟 Glob/Grep 同模型"。**

## 5. 关键决策（含考虑过的备选方案）

### 5.1 Snapshot 机制：复刻 Claude Code 原生

**决策**：SSHConnection 建立时执行一次 `bash -ic 'declare -p; declare -fp; alias 2>/dev/null'`，把结果写到远程 `/tmp/rmcp-snapshot-<host>-<pid>.sh`；每次 Bash 调用 `source` 此文件。

**为什么**：通过实测确认 Claude Code 原生 Bash 用的就是这套方法（见命令 `ps -p $$ -o args` 输出里的 `source /home/lb/.claude/shell-snapshots/snapshot-bash-*.sh`）。目标章节"对齐原生体验"要求我们复刻它。

**考虑过的备选方案**：

| 备选 | 优点 | 缺点 | 为什么不选 |
|------|------|------|----------|
| **A**：`bash --norc --noprofile -c <cmd>` | 最简单 | 丢失用户 PATH、aliases、conda init、`module load` 等环境 | 对集群/科研用户体验大幅下降；与 CC 原生不一致 |
| **B**：每次调用都 `bash -ic 'source bashrc; <cmd>'`（每次源 bashrc） | 简单，环境完整 | 每次都付 bashrc 启动开销（可能 100-500ms） | 性能不可接受 |
| **C（采用）**：snapshot + replay | 一次性付 bashrc 启动；后续每次便宜地 source snapshot；与 CC 原生一致 | 实现复杂；需 snapshot 文件生命周期管理 | 收益最大、与 CC 对齐 |

**Snapshot 创建命令**（在 SSHConnection.connect() 末尾执行）：

```bash
bash -ic 'declare -p 2>/dev/null; declare -fp 2>/dev/null; alias 2>/dev/null' \
  > /tmp/rmcp-snapshot-<host_name>-<server_pid>.sh
```

- `-i` interactive 让 `~/.bashrc` 被 source
- `-c` 一次性运行
- `declare -p` 导出全部已 declare 的变量（含 export/readonly 标记）
- `declare -fp` 导出全部函数定义
- `alias` 导出全部别名
- `2>/dev/null` 抑制无关警告
- snapshot 路径含 `<host_name>` 和 `<server_pid>`，避免多个 MCP 实例冲突

**Snapshot 使用命令**（每次 Bash 调用包装）：

```bash
bash --noprofile --norc -c "source /tmp/rmcp-snapshot-<host>-<pid>.sh 2>/dev/null || true; <user_cmd>" </dev/null
```

- `--noprofile --norc` 不再 source rc 文件（snapshot 已经包含其结果）
- `source ... 2>/dev/null` **关键**：抑制 source 时的 stderr 噪音——`declare -p` 会 dump 出 `BASH_VERSION`、`BASHOPTS`、`SHELLOPTS` 等 readonly 变量，source 时它们触发 `bash: line N: BASH_VERSION: readonly variable` 的报错；统一吞掉这类无害噪音
- `... || true` snapshot 文件不存在时退化继续（容错）
- `</dev/null` stdin 重定向到 /dev/null（解决 srun 类问题，与 CC 原生一致）

**snapshot 内容写入方式**：用 SFTP 直接写远程文件（`sftp.file(path, "w").write(content)`），**不**用 `exec("cat > path <<EOF")`——bashrc 导出内容含大量引号、`$`、反引号，here-doc 容易转义出错。`_create_snapshot()` 内部依赖 `connection.get_sftp()` 已就绪的 SFTP channel。

**snapshot 路径中 `<server_pid>` 的语义**：指**本地 MCP server 进程的 PID**（`os.getpid()`），不是远程 bash 的 PID——同一台远程主机被多个本地 MCP 实例连接时本地 PID 必然不同，可避免文件冲突。跨客户端机器的 PID 巧合碰撞概率极低（且不同客户端的 MCP 进程也不会同时清理同一文件），不加随机串。

**Snapshot 生命周期**：

- 创建：`SSHConnection.connect()` 末尾，建立 SSH 后立即创建
- 重建：`SSHConnection._do_reconnect()` 后重新创建（远程 `/tmp` 可能仍有旧 snapshot 但稳妥起见重建）
- 清理：`SSHConnection.close()` 中 `rm -f` 删除；进程异常退出时依赖 `/tmp` 重启清理

**失败容错**：

- 若 snapshot 创建失败（用户 bashrc 报错、`/tmp` 不可写、网络抖动等），**记 warning 但不致命**——后续 Bash 调用退化为不带 snapshot 的纯 `bash --noprofile --norc -c`。
- agent 仍能用，只是 PATH 等不含用户自定义。文档明说这个降级行为。

### 5.2 Cwd 不跨调用持久（命令内 `cd` 不存活）

**决策**：每次 Bash 调用都从**配置的 cwd**（见 §6）开始；agent 在 `command` 字符串里 `cd` 别处只在该次调用内生效，下一次 Bash 调用又回到配置的 cwd。

**为什么**：

1. **与 CC 原生一致**——实测原生 Bash 工具 `cd /tmp` 后下一调 `pwd` 回到项目根（每次都是 fresh shell）
2. **降低 agent 心智负担**——agent 不需要在心里维护"我上次 cd 到哪了"
3. **强迫 agent 使用 `cd dir && cmd` 的复合形式**——这是 CC 原生用户的标准模式

**与 v0.1.x 的对比**：

| 行为 | v0.1.x（持久 bash） | v0.2.0（非持久 + 配置 cwd） |
|------|-----------------|------------------------|
| 起始 cwd | sshd 默认（通常 `$HOME`） | 配置的 cwd（默认 `$HOME`，可 `--cwd /opt/app` 改） |
| `cd` 是否跨调用持久 | 是 | 否 |
| 输出 prefix/suffix | `[host=X cwd=<runtime pwd>]` 前缀 | `[host=X cwd=<configured>]` 后缀（§6.7） |

**输出格式变化挪到 §6.7 统一讨论**（不只 Bash，所有工具都加同一后缀）。

### 5.3 Timeout 用 `channel.close()` 而非 PTY Ctrl-C

**决策**：Bash 前台模式超时时调用 `channel.close()`；远程 sshd 检测到 channel 关闭后会给远程命令的 session 发 SIGHUP。

**为什么**：

1. **不需要 PTY**——v0.1.x 引入 PTY 的唯一理由就是让 `\x03` 生效。non-persistent 模型下 channel.close() 直接干掉远程进程，不需要 PTY 黑魔法
2. **更干净**——SIGHUP 比 SIGINT 更"决绝"；不会出现"命令吞掉 SIGINT 继续运行"的情况（如果用户命令显式 trap'd SIGINT 的话）
3. **更彻底**——SIGHUP 会传播到 session 内所有子进程（不像 SIGINT 只到前台进程组），所以命令派生的子进程也会被清理

**Background bash 不受影响**：因为它用 `setsid nohup`，进程脱离了 channel 的 session，channel 关闭也不影响后台进程——这正是 background 模式想要的行为。

### 5.4 Partial output 流式收集

**决策**：超时时返回**已经从 channel 收到的所有 stdout**，附在 `Error: Command timed out` 之前。

**实现方式**（伪代码仅演示流程，实际实现宜用 `channel.settimeout(...)` 让 `recv` 阻塞到数据或超时，避免 busy-poll 的 CPU 开销和小命令延迟）：

```python
# 简化伪代码（实际实现见 plan）
stdin, stdout, stderr = client.exec_command(wrapped_cmd, timeout=...)
out_buf = b""
deadline = time.time() + timeout
while not stdout.channel.exit_status_ready():
    if time.time() > deadline:
        # Timeout: drain whatever's available, then close
        while stdout.channel.recv_ready():
            out_buf += stdout.channel.recv(4096)
        stdout.channel.close()
        return f"{out_buf.decode()}\n\nError: Command timed out after Ns on X"
    if stdout.channel.recv_ready():
        out_buf += stdout.channel.recv(4096)
    else:
        time.sleep(0.05)
# Normal exit: read remaining, return with exit code
out_buf += stdout.read()
return ...
```

（注意伪代码已去掉旧的 `[host=X]\n` 前缀——后缀由 `server.py::call_tool()` 统一注入，见 §6.7。）

**为什么**：

1. 上一轮 agent feedback 里 Bug 2 的核心痛点就是 timeout 丢 stdout
2. Per-call exec 模型下这是**自然实现**——不像持久模型要靠 sentinel 协议绕一圈

### 5.5 Snapshot 与 Reconnect 的交互

**决策**：`SSHConnection._do_reconnect()` 流程末尾**重建 snapshot**——即新建一个 `/tmp/rmcp-snapshot-<host>-<pid>.sh`（PID 不变所以路径相同，会覆盖旧文件）。

**为什么**：

1. SSH 断了重连后，远程 `/tmp` 文件可能还在（只是连接换了）。理论上不重建也能用。
2. 但用户的 bashrc 可能在断线期间被修改；重建 snapshot 让 agent 总是看到最新环境，是更稳妥的语义。
3. 重建是一次性开销（~100-500ms），重连本来就是异常路径，多这点开销可接受。

### 5.6 Reconnect WARNING 简化

**决策**：reconnect 后的 WARNING 文本从 v0.1.x 的"shell state was reset; cwd is now $HOME, env vars lost, use absolute paths" 简化为"SSH connection to `<host>` was lost and has been re-established. Snapshot was rebuilt; if your bashrc has changed since the connection started, the new state takes effect from this point."

**为什么**：v0.1.x 那段警告假定了"agent 有累积的 shell 状态需要重新建立"。v0.2.0 没有这种累积状态了——每次调用本来就是 fresh shell。所以 WARNING 不再需要"教 agent 重做 setup"，只需要"通知 agent 链路有过中断"。

## 6. Cwd 配置功能（新增）

v0.2.0 的第二项主要变更，与"非持久 Bash"独立但同期落地。本节自成一体，可独立审查。

### 6.1 目标

让所有工具的相对路径有一个**明确的远程锚点**，对齐 Claude Code 原生"在本地项目目录下工作"的体验：

- 原生：agent 在 `/home/lb/workspace/myapp` 下启动 Claude Code → `Read("config.yaml")` 解析到 `/home/lb/workspace/myapp/config.yaml`
- v0.1.x：agent 调 `Read("config.yaml")` → 报错 `File not found`（必须传绝对路径），心智不一致
- v0.2.0：MCP 注册时 `--cwd /opt/myapp` → `Read("config.yaml")` 解析到 `/opt/myapp/config.yaml`，行为与 CC 原生一致

### 6.2 配置方式

**CLI 参数**（最高优先级）：

```bash
python -m remote_mcp --host prod --cwd /opt/myapp
```

**config.yaml 字段**：

```yaml
hosts:
  prod:
    hostname: prod.example.com
    user: deploy
    cwd: /opt/myapp     # 可选；支持前导 ~（连接后展开为远程用户 $HOME）
  staging:
    hostname: staging.example.com
    user: deploy
    cwd: ~/work/staging # ~ 展开为 /home/deploy/work/staging
```

**优先级**：CLI `--cwd` > `hosts.<name>.cwd` > 默认（远程 `$HOME`）

**默认行为**：未配置 cwd 时等价于 `cwd: ~`——使用远程用户 home。这保持 v0.1.x 的工作目录行为不变（仅"相对路径不再报错"是新增）。

**cwd 字段合法值**（不论来源是 CLI 还是 YAML）：

- **绝对路径**：以 `/` 开头（例：`/opt/app`、`/`）
- **用户 home**：精确等于 `~` 或以 `~/` 开头（例：`~`、`~/projects/myapp`）

其他形式**启动时 fail-fast**：

```
Error: cwd must be an absolute path or start with '~/' (got: '<value>').
       Valid:   /opt/app, ~/projects/myapp, ~
       Invalid: myapp        (裸相对路径)
                ./projects   (startup-cwd 相对，不接受)
                ../app       (同上)
                ~root        (他人 home；只接受当前 SSH 用户的 home)
                ~user/foo    (同上)
```

实现层校验：`path == "~" or path.startswith("/") or path.startswith("~/")`——以"`~` 后必须紧跟 `/` 或字符串结尾"为准，拒绝 `~user` 这类前缀字符 `~` 但非 home 缩写的输入。

**`~` 展开时机和写回**：`SSHConnection.connect()` 末尾、SFTP stat 校验之前，执行一次 `bash -c 'echo $HOME'` 拿到远程用户 home（用 `-c` 而非 `-ic`：`$HOME` 由 sshd 在登录前 export，不需要 bashrc，避免无谓启动开销），替换 cwd 字段里的前导 `~`；展开结果**写回 `self.config.cwd`**（覆盖原始字段值）——这样后续 `_create_snapshot()` 里的 `shlex.quote(self.config.cwd)`、RemoteInfo 输出、统一后缀里展示的 cwd 都是同一个绝对路径，避免歧义。一次性开销。

### 6.3 路径解析规则

新增模块 `remote_mcp/paths.py`：

```python
import posixpath

def resolve_path(path: str, cwd: str) -> str:
    """工具传入的路径解析为远程绝对路径；相对路径基于 cwd 锚定。

    Raises ValueError on invalid input.
    """
    if not path:
        raise ValueError("empty path")
    if path.startswith("~"):
        raise ValueError(
            "path starts with '~' — use an absolute path, "
            "or a path relative to the configured cwd"
        )
    if path.startswith("/"):
        return path  # 绝对路径原样返回
    return posixpath.normpath(posixpath.join(cwd, path))
```

| 输入 | cwd=`/opt/app` 时的结果 | 备注 |
|------|---------------------|------|
| `"/etc/passwd"` | `/etc/passwd` | 绝对路径不动 |
| `"foo.txt"` | `/opt/app/foo.txt` | 标准相对 |
| `"./foo.txt"` | `/opt/app/foo.txt` | `./` 由 normpath 吸收 |
| `"../sibling"` | `/opt/sibling` | **允许出 cwd**（无沙箱，对齐 CC 原生） |
| `"."` | `/opt/app` | 等价于 cwd 本身 |
| `""` | `ValueError("empty path")` | 防止意外行为 |
| `"~/foo"` | `ValueError("path starts with '~'...")` | 工具参数里不允许 |

**两层 `~` 政策的对比**（同一个符号，两层语义）：

- **cwd 配置字段**：支持 `~`（连接时展开为远程 $HOME）—— 是"用户写配置时的便利"
- **工具参数**：拒绝 `~`（报错）—— agent 不应假设服务器 $HOME 的值；要么传绝对路径，要么传相对 cwd 的路径

工具层 `resolve_path` 抛 `ValueError` 后，每个工具自己捕获并转成 `"Error: ..."` 字符串返回（与现有错误处理风格一致；工具不能抛异常）。

### 6.4 启动时校验：fail-fast

`SSHConnection.connect()` 末尾、snapshot 创建之前，对配置的 cwd 做存在性检查。

**前置依赖**：本节假定 `_resolve_remote_home()` 已经跑过，`self.config.cwd` 此时已是绝对路径（不再含前导 `~`）。SFTP `stat` 不展开 `~`，把 `~/foo` 直接喂进去会失败——所以"先展开再 stat"的顺序不可逆（流程见 §6.9）。

**用 SFTP `stat`，不用 `bash test -d`**——SFTP 通道不依赖远端 shell 是否正常（restricted shell、sshd ForceCommand 等场景下 `test` 都可能不可达），更稳：

```python
import stat as _stat
sftp = self.get_sftp()
try:
    st = sftp.stat(cwd)
except IOError:
    raise SSHConnectionError(
        f"configured cwd '{cwd}' does not exist on host '{self.config.name}'. "
        f"Fix the --cwd argument or remove it to use $HOME."
    )
if not _stat.S_ISDIR(st.st_mode or 0):
    raise SSHConnectionError(
        f"configured cwd '{cwd}' exists on host '{self.config.name}' but is "
        f"not a directory. Fix the --cwd argument."
    )
```

**为什么 fail-fast 而不是退化**：cwd 是用户**显式**配置的——他在 `claude mcp add` 里写了 `--cwd /opt/app`，意思是"我要工作在那里"。如果路径不存在，是配置错误，应当立即可见而非让每个工具调用各报各的错。

**未配置 cwd（默认 `~`）时**：仍走 SFTP `stat` 路径（校验 `$HOME` 存在并且是目录）——保险但开销可以忽略。

### 6.5 Bash 的 cwd 实现：写进 snapshot

**不**在每次调用的 wrap 里加 `cd`，而是在 snapshot 文件末尾**追加一行** `cd <cwd> || exit 1`。这样 snapshot 自带 cwd 设置，per-call wrap 保持简单。

修改 `_create_snapshot()`：

```python
content = self.exec(
    "bash -ic 'declare -p 2>/dev/null; declare -fp 2>/dev/null; alias 2>/dev/null'"
).stdout
# Append cwd cd at the very end of the snapshot (cwd already ~-expanded and validated)
content += f"\ncd {shlex.quote(self.config.cwd)} || exit 1\n"
self._write_remote(self._snapshot_path, content)
```

`|| exit 1` 是保险——connect 时虽已校验过 cwd 存在，但万一中途被删，bash 子进程会立即退出而不是在错的 cwd 跑命令（命令会得到非 0 exit code 与空 output；agent 看到错误比看到"在错目录的成功"安全得多）。

Per-call wrap **保持简单**（snapshot 自己处理 cd）：

```bash
bash --noprofile --norc -c "source /tmp/rmcp-snapshot-<host>-<pid>.sh 2>/dev/null || true; <user_cmd>" </dev/null
```

注意 `source ... || true` 仍是容错（snapshot 失败时退化继续）；但 snapshot 内的 `cd ... || exit 1` 不被这条 `|| true` 影响——`cd` 失败让整个 bash 退出，wrap 里的 `<user_cmd>` 根本不会被执行。

### 6.6 其他工具的 cwd 实现：Python 层 resolve_path

Read/Write/Edit/MultiEdit/MultiRead/FileStat/Glob/Grep/Upload/Download 都走 SFTP 或一次性 exec_command——没有 shell 状态承载，所以在 Python 层用 `resolve_path(path, conn.config.cwd)` 处理。

| 工具 | 集成方式 | 备注 |
|------|--------|------|
| Read | `file_path = resolve_path(file_path, conn.config.cwd)` | 1 行 |
| Write | 同上 | 1 行 |
| Edit | 同上 | 1 行 |
| MultiEdit | 同上 | 1 行 |
| MultiRead | for-loop 内对每个 `reads[].file_path` 解析 | 循环 |
| FileStat | 对 `file_paths`（str 或 list）逐项解析 | 循环 |
| Glob | 解析 `path` 参数（默认 `"."` → cwd）；`pattern` 不动 | 1 行 |
| Grep | 解析 `path` 参数（默认 `"."` → cwd）；`pattern` 和 `include` 不动 | 1 行 |
| Upload | 仅解析 `remote_path`；`local_path` 仍按本地语义（`os.path.expanduser`） | 不对称：local 走 OS 标准，remote 走 cwd |
| Download | 同上 | 同上 |
| Bash | **不**在 Python 层解析——shell 语义自处理（snapshot 已 cd 到 cwd） | 见 §6.5 |
| Feedback | 无路径参数 | 不动 |
| RemoteInfo | 无参数 | 但输出新增 `cwd` 字段（§6.8） |

**Glob `path="."` 语义新明确**：v0.1.x 隐含跑在 sshd 默认 cwd（行为未文档保证）；v0.2.0 解析到配置 cwd（行为明确可预期）。**输出从相对路径变为绝对路径**（与 CC 原生 Glob 一致）——见 §7 breaking changes 行。

**Upload/Download 的本地/远程不对称**是有意的——agent 调 `Upload(local_path="myfile.txt", ...)` 时，"myfile.txt 在哪"由 MCP server 进程的本地 cwd 决定，跟 remote cwd 无关。

**`resolve_path` 对绝对路径是 no-op**（见 §6.3 表）——工具内部无条件调用即可，绝对路径传进去原样返回，不需要先判断"path 是绝对路径就跳过 resolve"。

### 6.7 工具输出后缀统一

**所有工具的输出（成功和错误）一律追加后缀**：

```
[host=<host_name> cwd=<configured_cwd>]
```

格式规则：

- **后缀**（不是前缀）—— 避免被 Read 等带行号的输出误读为第 0 行
- **空行分隔**—— `<output>\n\n[host=X cwd=Y]\n`
- **显示配置的 cwd**—— 不查询运行时 pwd；agent 每次都看到稳定的"我在 X"心智模型
- **统一应用**—— 成功、错误、Warning（reconnect）输出都加

样例：

```
     1	#!/bin/bash
     2	echo hello

[host=prod cwd=/opt/app]
```

```
Error: File not found: foo.txt

[host=prod cwd=/opt/app]
```

```
[WARNING] SSH connection to prod was lost and has been re-established. Snapshot was rebuilt; if your bashrc has changed since the connection started, the new state takes effect from this point.

<tool's normal output>

[host=prod cwd=/opt/app]
```

**实现位置**：`server.py::call_tool()` dispatch 层统一处理——每个工具返回纯字符串（成功或 `Error: ...`），`call_tool` 拿到字符串后追加后缀。**工具自身不实现后缀**，避免 13 个工具各写一遍。同时 `tools/bash.py:59,105` 现有的 `[host=X cwd=Y]\n` 前缀拼接**必须随 Bash 重写删除**（§4.2 已列），否则会出现"前缀 + 后缀"双 host 标签。

**完整算法**（含 reconnect WARNING 前缀的拼合顺序）：

```python
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    result = _with_retry(lambda: _raw_dispatch(name, arguments))

    # Step 1: reconnect WARNING 前缀（v0.1.x 已有，文本由 §5.6 简化）
    prefix = ""
    if _conn is not None and _conn.check_and_clear_reconnect_flag():
        prefix = (
            f"[WARNING] SSH connection to {_conn.config.name} was lost "
            f"and has been re-established. Snapshot was rebuilt; if your "
            f"bashrc has changed since the connection started, the new "
            f"state takes effect from this point.\n\n"
        )

    # Step 2: 统一后缀（v0.2.0 新增）
    suffix = f"\n\n[host={_conn.config.name} cwd={_conn.config.cwd}]"

    # Step 3: 拼合 — WARNING 在最前，主体在中间，host/cwd 后缀在最后
    return [TextContent(type="text", text=prefix + result + suffix)]
```

最终输出结构（每段之间用 `\n\n` 空行分隔）：

```
[WARNING] ... (可选，仅 reconnect 后第一次调用出现)

<tool 主体输出>

[host=X cwd=Y]
```

**Reconnect 失败路径**：`_with_retry` 重连失败时返回字符串 `"Error: SSH connection to <host> lost and reconnect failed: <reason>"` 作为 `result`，`check_and_clear_reconnect_flag()` 返回 False（无成功重连），所以 `prefix` 为空。**后缀照常附加**——`_conn.config.cwd` 是配置值（不依赖 transport 存活），渲染后缀只是带上上下文标签，方便 agent 知道"是哪台主机出了问题"。最终输出形如：

```
Error: SSH connection to prod lost and reconnect failed: <reason>

[host=prod cwd=/opt/app]
```

这是预期行为；agent 看到 `Error: ... reconnect failed` 即知连接死了，后缀的 cwd 仅用于"我刚才是在跟哪台失败的"识别。

**为什么不用"教学式提示"**（曾考虑过的替代）：原方案是 Bash 在 `pwd != cwd` 时显示"你刚才 cd 了别处，下次会重置"。否决理由：提示应当**避免**错误而非**事后**解释。统一后缀让 agent 每次都看到固定 cwd，自然不会产生"cd 还在"的错觉，比事后纠正更主动。

**副效应**：v1 spec 提到的 sentinel `_CWD_$(pwd)` 部分**不再需要**——cwd 已知（配置死的），运行时 pwd 不暴露给 agent。本来 v0.2.0 整个 sentinel 协议都废弃（见 §4.2），所以这条只是确认"也不需要新机制去抓 pwd"。

### 6.8 RemoteInfo 输出新增 cwd 字段

v0.1.1 引入的 RemoteInfo 工具输出新增 `cwd` 字段，让 agent 能在不调 `Bash("pwd")` 的情况下查到当前生效的 cwd。**格式保持 v0.1.x 的 `key=value`**（不改成 `key: value`，与现有 `remote_info.py` 一致），仅追加一行：

```
host=prod
user=deploy
hostname=prod.example.com
port=22
jump_host=bastion
cwd=/opt/myapp

[host=prod cwd=/opt/myapp]
```

字段加在 `jump_host` 之后、统一后缀（空行后）之前。`cwd` 值是**`~` 展开后**的绝对路径（见 §6.2 中"`~` 展开结果写回 `self.config.cwd`"）。

### 6.9 与 Snapshot/Reconnect 的交互

**正常 connect 流程**：

```
1. paramiko SSH connect
2. 校验 cwd 字段格式（必须 / 或 ~ 或 ~/ 开头，否则 fail-fast；§6.2）
3. 查询远程 $HOME（bash -c 'echo $HOME'，用于 ~ 展开；§6.2）
4. cwd ~ 展开（如有），结果写回 self.config.cwd
5. SFTP stat <cwd> 校验存在且为目录（fail-fast；§6.4）
6. 创建 snapshot（含 cd <cwd> || exit 1 末尾追加；§5.1、§6.5）
7. 准备就绪，接受工具调用
```

**Reconnect 流程**：与 connect 完全一样——snapshot 重建时 cwd 行自然包含在新 snapshot 里。`~` 展开也重做（连接到的远程 $HOME 理论上不变，但用一致流程更稳）。

**Cwd 与 snapshot 的概念分工**：

- snapshot 装载用户**环境**（PATH、aliases、functions、export 的变量）
- cwd 设定**起始目录**
- 二者都通过同一个 snapshot 文件交付给每次 bash 调用

## 7. Agent-visible Breaking Changes

按 agent 实际能观察到的变化列出，agent 训练数据如果依赖这些行为会失效：

| 变化 | v0.1.x 行为 | v0.2.0 行为 | Agent 应当如何适应 |
|------|-----------|-----------|------------------|
| cwd 跨调用 | `cd /tmp` 后下一调 `pwd` 是 `/tmp` | `cd /tmp` 后下一调 `pwd` 是配置 cwd（默认 `$HOME`） | 用 `cd /tmp && cmd` 单次拼接 |
| env 跨调用 | `export FOO=bar` 后下一调 `echo $FOO` 是 `bar` | 是空 | 用 `FOO=bar cmd` 内联，或 `bash -c "export FOO=bar; cmd"` 单次包装 |
| source venv | `source venv/bin/activate` 后下一调 `python` 用 venv | `python` 还是系统 python | `venv/bin/python script.py` 绝对路径，或 `. venv/bin/activate && python ...` 单次拼接 |
| ssh-agent | `eval $(ssh-agent); ssh-add` 后下一调 `git pull` 用 agent | 不再有 agent；下一调失败 | 用无 passphrase 的 deploy key，或单次内 setup+use 拼接 |
| 工具输出格式 | `[host=X cwd=<runtime>]\n<output>` 前缀，仅 Bash 有 | `<output>\n\n[host=X cwd=<configured>]` 后缀，**所有工具**都有 | 解析输出的 agent 脚本若按行号/位置取数据，需要忽略末尾的后缀行 |
| 工具支持相对路径 | 所有工具 path 参数必须绝对路径 | path 参数可相对（基于配置 cwd 解析） | 新能力，向后兼容；agent 可继续传绝对路径，或开始用相对路径 |
| `~` 在工具 path 参数 | 未明确（依赖底层 shell/SFTP 行为，多数情况下不展开） | 显式报错 `Error: path starts with '~' ...` | 传绝对路径或相对 cwd 的路径；不要用 `~/foo` |
| Glob `path="."` 语义 | 跑在 sshd 默认 cwd（未文档化） | 跑在配置 cwd（明确） | 行为现在可预期；旧脚本无 contract 保证，影响很小 |
| Glob/Grep 默认输出路径形式 | 相对路径 `./foo.py`（默认 `path="."` → `find . ...`）| 绝对路径 `/opt/app/foo.py`（默认解析后 `find /opt/app ...`）| **正向 breaking** —— 与 CC 原生 Glob 行为一致（CC 原生也返绝对路径）；agent 拿到绝对路径后可直接喂给 Read/Edit/Write 无需再拼，反而更方便 |
| Reconnect WARNING 文本 | 详细的"state was reset"指引 | 简化为"connection was reset" | agent 不需要做特殊事情；旧的"重做 setup"建议不再适用但也无害 |
| 单次 Bash 调用延迟 | < 50ms typical | 多 100-500ms（VPN）或 500-1000ms（慢 FS） | agent 倾向于把多个相关命令用 `&&` 拼成单次调用，减少调用次数 |

## 8. 接受的取舍

### 8.1 性能代价

**每次 Bash 调用比 v0.1.x 慢 100-1000ms**（具体值看 RTT 和远程 FS 速度）。

- 局域网（RTT 1ms）+ 本地 FS：+50-150ms（主要是 bash 进程启动）
- 普通 VPN（RTT 50-200ms）：+200-600ms
- 跨洲 VPN（RTT 500ms+）+ 慢共享 FS：+1-2s

**为什么接受**：

1. 单次延迟可量化、可文档化
2. CC 原生在同样条件下付同样代价；agent 设计本来就考虑了这点
3. agent 适应方式（多命令 `&&` 拼接）反而带来正面效果（减少总往返次数）
4. v0.1.x 的"无延迟"是 sentinel + PTY + reader thread + setsid + 一堆补丁堆出来的；削减复杂度本身是巨大收益
5. **future work** 已记录 opt-in `mode: persistent` 作为反向出口——如果将来有用户对慢 FS 实在不能容忍，可重新启用持久模式

### 8.2 部分工作流被破坏

ssh-agent / conda activate / nvm use / source venv 这类"设状态供后续使用"的命令，跨调用不再生效。

**为什么接受**：

1. 这些 use case 在远程 agent 场景里实际频率低
2. 全部都有可接受的 workaround（内联 setup+use、绝对路径）
3. CC 原生用户已经适应这种模式
4. CLAUDE.md.fragment.md 会显式给出替代写法，agent 学习成本不高

### 8.3 第一次连接 + snapshot 创建多 ~500ms

`SSHConnection.connect()` 末尾的 snapshot 创建是 `bash -ic` 启动 + dump，远程 bashrc 复杂的话可能 100-500ms。再加 §6 引入的 `bash -c 'echo $HOME'`（~ 展开）+ SFTP `stat <cwd>`（fail-fast）各 1 个 RTT。

**为什么接受**：

1. 一次性开销，被整个 MCP server 生命周期摊薄
2. Reconnect 时重建多付一次，但 reconnect 本身就是异常路径
3. fail-fast 校验代价远低于"agent 在错的 cwd 跑工具排错"的代价

### 8.4 每次工具调用输出多出后缀一行

所有工具输出末尾固定 `\n\n[host=X cwd=Y]\n`（~30-50 字节）。

**为什么接受**：

1. 短工具调用（如 `Bash("ls")`）的相对开销 ~5%；长输出（Read 大文件）可忽略
2. 这是"防 agent cwd 心智失稳"的核心机制（§6.7），便宜的预防成本
3. agent 解析自然忽略（看的是主输出，后缀只是上下文）

## 9. 明确不做（指向 future work）

| 项 | 为什么不做 | 在哪里记录 |
|---|----------|---------|
| 流式输出（agent 边跑边看 stdout） | MCP 协议层还没支持 | v1 spec §15.2（已于本次讨论中追加） |
| Opt-in `mode: persistent` 配置 | YAGNI——等真用户痛了再做 | v1 spec §15.2（已追加） |
| 跨调用 cwd 持久（即使非持久 bash 也可以"假装" cwd 持久） | CC 原生不做；做了会与目标"对齐原生"冲突 | 本规范 §5.2 |
| `interactive: true` 参数让 stdin 不重定向 /dev/null | YAGNI；CC 原生没这个；真有需要用 `Bash + tmux send-keys` 模式 | 本规范不实现 |
| 自动重试 snapshot 创建失败 | 降级到无 snapshot 已经够用 | 本规范 §5.1 容错小节 |
| 多 cwd / `--add-dir` 等价物（CC 原生有） | YAGNI；单 cwd 已覆盖 95% 用例；绝对路径作 escape hatch | 本规范不实现 |
| 工具参数支持 `~` 展开 | agent 不应假设服务器 $HOME；明确报错让 agent 用绝对路径或相对 cwd | 本规范 §6.3 |
| cwd 沙箱（限制 agent 不能 `..` 出 cwd） | CC 原生不做；security boundary 在 SSH 用户权限层而非 MCP 层 | 本规范 §6.3 |

## 11. 验收标准

按"agent 视角"和"开发者视角"分两组：

### 11.1 Agent 视角（黑盒）

**非持久 Bash 相关**：

- [ ] **`mcp__remote-<host>__Bash("echo hello")`** 返回 `hello\n\n[host=<host> cwd=<configured-cwd>]\n`
- [ ] **`Bash("cd /tmp")` 后** `Bash("pwd")` 返回配置 cwd（不是 `/tmp`）—— cd 不跨调用持久
- [ ] **`Bash("export FOO=bar")` 后** `Bash("echo $FOO")` 返回空 —— env 不持久
- [ ] **`Bash("srun -p debug6 -A penglin -N1 -n1 hostname")`** 在合理时间内返回带 hostname，不再永久挂起
- [ ] **`Bash("cat")` (无参数读 stdin)** 立刻返回（EOF），不再永久挂起
- [ ] **`Bash("echo START; sleep 999", timeout=2)`** 返回包含 `START` + `Error: Command timed out after 2s` —— partial output 不丢
- [ ] **`Bash("which python")`** 找到用户 PATH 里的 python（snapshot 起作用）

## 附录：术语

- **持久 bash session**：v0.1.x 模型——一个 bash 进程通过 PTY 接收命令注入，跨工具调用复用。**v0.2.0 已废弃**。
- **non-persistent bash**：v0.2.0 模型——每次 Bash 工具调用都是一个独立的 `exec_command` channel，bash 进程一次性。
- **snapshot 机制**：在远程主机上一次性导出 bashrc 加载后的 shell 环境到一个文件，后续每次 Bash 调用 `source` 该文件——既保留用户环境又避免每次启动 bashrc 开销。v0.2.0 中 snapshot 末尾还追加 `cd <cwd>` 以承担 cwd 设置职责。
- **per-call channel**：每次工具调用都新开一个 SSH channel，命令结束后关闭——与 Glob/Grep 等工具的现有模式相同。
- **配置 cwd / configured cwd**：通过 `--cwd` 或 `hosts.<name>.cwd` 指定的远程工作目录。所有工具的相对路径以此为锚。默认是远程用户 `$HOME`。
- **resolve_path**：`remote_mcp/paths.py` 的核心 helper，把工具参数里的路径（绝对/相对/`.`/`..`/空/`~` 等）映射为远程绝对路径或可被 agent 理解的错误。
- **统一后缀**：所有工具返回值末尾追加的 `[host=X cwd=Y]` 一行（用空行与主输出分隔），由 `server.py::call_tool()` 统一注入，用于稳定 agent 的"我在哪台主机的哪个目录"心智。
