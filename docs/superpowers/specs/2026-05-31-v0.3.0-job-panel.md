# remote-mcp v0.3.0 设计规范：后台任务面板

**日期**：2026-05-31
**状态**：待审核 → 待实施
**前身**：增量于 [`2026-05-26-remote-mcp-design.md`](./2026-05-26-remote-mcp-design.md)（v1）、[`2026-05-27-v0.2.0-non-persistent-bash.md`](./2026-05-27-v0.2.0-non-persistent-bash.md)（v0.2.0/v0.2.1）、[`2026-05-28-network-robustness-design.md`](./2026-05-28-network-robustness-design.md)（v0.2.2/v0.2.3）。本规范覆盖 v0.3.0 的"后台任务面板"子系统：扩展 `Bash(run_in_background=True)` 的参数与返回结构，并新增 4 个面板管理工具。其余设计（连接生命周期、SFTP 工具、非持久前台 Bash、cwd 配置、网络鲁棒性契约等）继续以前述 spec 为准。

---

## 1. 概述与目标

v0.2.x 把 `Bash(run_in_background=True)` 做到了正确——`setsid + nohup + </dev/null + &` 这套防 SIGHUP / 防 stdin 占用 / 防 shell 退出的零件齐全；远端进程在 SSH 断开后仍能存活；pidfile 保证响应丢失时 PID 可找回。但实际使用中，agent 在管理多小时长任务时仍倾向手写 `nohup` 而非使用该工具。原因有两个：

1. **工具描述未告知 agent 实际生成的 shell 命令**。Agent 不知道 `run_in_background=True` 已经包含 setsid + nohup，因此不放心把 18 小时的 build 托付给它，宁可手写 nohup 以"亲眼看到 SIGHUP 防御"。这是文档可发现性问题，而非功能问题。
2. **不支持命名、列表、kill、状态查询等管理操作**。Agent 必须自己用 `pgrep`/`tail`/`kill` 组合每次 30+ 行 boilerplate 才能查询任务状态——量化损失在 18 小时 build 场景下约 75 KB token / session（详见 §3 动机）。

v0.3.0 解决这两个问题：

- **改写 `BASH_DESC`**，把 foreground 与 background 实际生成的 shell 命令展开告知 agent；顺便修正 v0.2.0 残留的"shell state persists across foreground calls"错误描述。
- **扩展 `Bash(run_in_background=True)`** 的参数（`log_path` / `name`）与返回结构（id / name / log_path / pid / started_at）。
- **新增 4 个面板管理工具**：`Jobs`（查询）、`JobKill`（kill）、`JobArchive`（归档）、`JobScript`（挂载状态脚本）。
- **不新增**独立的 `RemoteSpawn` 工具——启动入口仍是 `Bash(run_in_background=True)`，避免破坏 agent 既有习惯。
- **不新增**独立的 `JobStatus` 工具——其聚合查询能力以"每任务可挂载的状态脚本"形式融入面板。

总工具数从 13 增加到 17。

## 3. 动机

### 3.1 Agent feedback 的两条原始反馈

来自 2026-05-28 同一 session 内 35 秒间隔提交的两条 enhancement feedback（agent 跑 18 小时 LLVM/Python build 后所写），核心数据：

**反馈 #1（JobStatus）**：每次任务状态查询需手写 30+ 行 boilerplate：

```bash
pgrep -af login_.*_python.sh 2>&1
echo "===build.log==="
ls -la ~/tmp/python_build/logs/python.build.log 2>&1
echo "===tail build==="
tail -3 ~/tmp/python_build/logs/python.build.log 2>&1
echo "===login.out tail==="
tail -5 ~/tmp/python_build/login.out 2>&1
echo "===install python3?==="
ls ~/install/Python-3.13.4/bin/python3 2>&1 || echo NOT_YET
```

单次返回 1-2 KB（其中大部分是 locale "总用量" 文本、tail 多行原文、pgrep 完整命令行）。一晚 50+ 次 check 累计 ~75 KB token，其中 99% 用来表达"还在跑"这一个 bit 的信息。

**反馈 #2（RemoteSpawn）**：长 build 全部用手写 nohup 启动而非 `run_in_background=True`，原因是：

- log 路径不可控（默认 `/tmp/rmcp-bg-<uuid>.log`，不适合 cron 持续监控固定路径）
- 多并发任务用随机 UUID 难以区分（手写时用脚本名 + 自选 log 路径区分）
- 跨 session 续接困难（笔记本休眠醒来后 UUID 丢失）

### 3.2 经过澄清后的真正缺口

与反馈 agent 沟通后澄清：**反馈 #2 三条理由的根本原因是 agent 不知道 `run_in_background=True` 已经包含 setsid + nohup**——agent 误以为该工具仅做"fire-and-forget exec"，因此为了 18 小时长 build 的安全宁可手写 nohup 以"亲眼看到 SIGHUP 防御"。Log 路径、命名、跨 session 续接的痛点在已知 mcp 后台机制完备的前提下，本可以通过工具描述指引 agent 用 `Bash(run_in_background=True, command="bash build.sh > ~/build.log 2>&1")` 加 `pgrep + 已知 log 路径` 的组合解决。

但**反馈 #1 的聚合查询缺口是真实结构性缺失**，无法靠文档解决——agent 没有别的途径在一次 SSH 调用内得到"进程是否还活着 + 进度 + 主日志尾巴 + 失败标记 + 完工 artifact"的结构化聚合。

因此 v0.3.0 的设计基于以下重定位：

- 反馈 #2 的"启动正确性"已由 v0.2.x 解决，只需文档暴露 → §13 `BASH_DESC` 改写
- 反馈 #2 的"log 路径可控、可命名"是 ergonomic 改进 → §5 Bash 新增 `log_path` / `name`
- 反馈 #2 的"跨 session 列表/kill"是真实管理缺口 → §8 / §10 / §11 新增面板工具
- 反馈 #1 的"聚合查询"是真实结构性缺失 → §9 面板支持挂载状态脚本

## 4. 架构总览

```
Claude Code
     │ stdio MCP
     ▼
remote-mcp (local Python, --host <name>)
 ├── server.py: list_tools / call_tool / dispatch / 后缀 / WARNING
 ├── connection.py: SSHConnection (paramiko, compress=True, snapshot)
 ├── jobs/: 新增模块
 │    ├── sid.py: 派生 session id（PPID + parent start_time）
 │    ├── registry.py: meta.json / pid / next_id 的 SFTP 读写
 │    ├── state.py: 远端 kill -0 + date 批量查询、状态机推导
 │    └── scripts.py: status.sh 的写入、运行、超时控制
 └── tools/
      ├── bash.py: 扩展 run_in_background 路径
      └── jobs.py / job_kill.py / job_archive.py / job_script.py: 新增
                │ SSH (paramiko Transport, compress=True)
                ▼
              remote host
                ~/.cache/remote-mcp/jobs/<sid>/
                ├── next_id
                └── <name>/
                     ├── meta.json
                     ├── pid
                     └── status.sh    (可选)
```

**关键边界**：

- 面板状态完全持久化在**远端**（`~/.cache/remote-mcp/jobs/<sid>/`），mcp server 进程内不缓存，每次查询按需重新从远端读
- mcp server 进程内**不开后台线程**——所有面板操作都是同步、有界时间的工具调用
- sid 派生于 mcp server 的父进程（Claude Code）的 PPID + 启动时间，保证：
  - 同一 Claude Code session 内 mcp server 崩溃重启 → sid 不变 → 面板保留
  - 不同 Claude Code session 并发 → 不同 sid → 不同子目录 → 互不干扰

## 5. Bash 工具扩展（`run_in_background=True` 路径）

### 5.1 Python 签名

```python
def bash(conn, command: str, *,
         run_in_background: bool = False,
         timeout: float = None,
         description: str = "",
         log_path: str = None,
         name: str = None) -> str:
```

所有新参数（`log_path` / `name`）以及现有可选参数全部 **keyword-only**（`*` 之后）。Foreground 路径（`run_in_background=False`）忽略 `log_path` 与 `name`；description 在两条路径下都是 CC native 兼容字段，foreground 时保持 v0.2.x 的"占位未使用"语义，background 时作为面板 description 持久化到 meta.json。

### 5.2 MCP schema（`schemas.py::BASH_SCHEMA`）

```json
{
  "type": "object",
  "properties": {
    "command": {"type": "string"},
    "description": {
      "type": "string",
      "default": "",
      "description": "Brief description of the command (CC native compat). When run_in_background=true, this is also stored as the task description in the panel (max 500 chars; longer is truncated with a notice in the return)."
    },
    "timeout": {"type": "number", "default": 120},
    "run_in_background": {"type": "boolean", "default": false},
    "log_path": {
      "type": "string",
      "description": "Optional. Absolute remote path for stdout+stderr redirection (background only). Defaults to /tmp/rmcp-bg-<uuid>.log. Parent dirs auto-created via mkdir -p. If a non-directory file exists at the parent path, Error returned and task not launched. If the target log file exists, it is overwritten (shell '>' semantics)."
    },
    "name": {
      "type": "string",
      "description": "Optional. Job alias for panel reference (background only). Defaults to 'bg-<uuid12>'. Must be unique among active (non-archived) jobs in the current session — collision returns Error. Pattern: [A-Za-z0-9_.-]+ length 1-64."
    }
  },
  "required": ["command"]
}
```

### 5.3 Background 启动语义

#### 5.3.1 参数校验（启动前同步）

| 检查 | 失败行为 |
|---|---|
| `name` 提供时正则 `^[A-Za-z0-9_.-]{1,64}$` | `Error: invalid job name 'X': must match ^[A-Za-z0-9_.-]{1,64}$` |
| `description` 长度 ≤ 500 字符 | 超长截断到 500 字符，返回里附加 `[description truncated to 500 chars]` 提示 |
| `name` 与活跃任务冲突（活跃 = 不在归档状态） | `Error: job name 'X' already in active panel; archive the old one with JobArchive(name='X') or pick a different name` |
| `log_path` 父路径已存在但是普通文件 | `Error: log_path parent '/foo/bar' exists but is not a directory; cannot mkdir -p` |
| `log_path` 自身是已存在目录 | `Error: log_path '/foo/bar' is a directory, not a file` |

`name` 未提供时自动生成 `bg-<uuid12>`（如 `bg-a1b2c3d4e5f6`），保证 12 字符 hex 唯一性足以避免实际冲突。

#### 5.3.2 启动顺序（任一步失败 → best-effort 清理 → Error 返回，**不 exec**）

1. **派生 sid**（详见 §12）；若 sid 派生失败，回退到 mcp 进程启动 UUID + 提示
2. **SFTP mkdir** `~/.cache/remote-mcp/jobs/<sid>/`（递归，已存在跳过）
3. **校验 name 唯一性**：SFTP stat `~/.cache/remote-mcp/jobs/<sid>/<name>/`；若存在且其 meta.json 的 `archived_at_unix` 为 null → 冲突错误
4. **SFTP mkdir** `~/.cache/remote-mcp/jobs/<sid>/<name>/`（若已存在则跳过——此时一定是同名归档任务的目录复用，subsequent steps 会 truncate-write 覆盖旧内容；旧任务的 meta.json 自此从 mcp 工具不可见，详见 §11.3）
5. **SFTP mkdir** `log_path` 父目录（若 `log_path` 已提供且父目录不存在）
6. **分配 id**（atomic）：单次 exec `flock ~/.cache/remote-mcp/jobs/<sid>/.id_lock -c 'n=$(cat ~/.cache/remote-mcp/jobs/<sid>/next_id 2>/dev/null || echo 0); echo $((n+1)) > ~/.cache/remote-mcp/jobs/<sid>/next_id; echo NEW_ID=$n'`，解析 `NEW_ID=N` 取得分配的 id。flock 保证同一锁文件内串行执行，同一 session 内多个并发 launch（asyncio 触发）安全。`.id_lock` 文件不存在则 flock 自动创建。
7. **SFTP write** `meta.json`（schema 详见 §6）；此时 `pid` 字段为 null（exec 还未发出）
8. **远端 exec** wrap（§5.3.3）

任一步失败时的清理：仅当步骤 4 刚刚创建了新 `<name>/` 目录（即步骤 3 之前 stat 显示不存在）且步骤 5+ 失败时，删除该目录及其下已写入的 meta.json / pid / 等文件。如果步骤 4 复用了已存在的归档目录，**不删除**——保留旧归档文件不破坏 §11.3 的归档承诺。清理本身用 best-effort try/except，失败不掩盖原 Error；原 Error 文本附加 `(partial files may remain at ~/.cache/remote-mcp/jobs/<sid>/<name>/)`。

#### 5.3.3 Wrap 模板

```bash
( setsid nohup bash --noprofile --norc -c '
    source <snapshot_path> 2>/dev/null || true
    <user command>
  ' > <log_path> 2>&1 </dev/null &
  PID=$!
  echo $PID > <jobs_dir>/<sid>/<name>/pid
  STARTED_AT_UNIX=$(date +%s)
  STARTED_AT=$(date -u -Iseconds)
  echo "BG_PID=$PID"
  echo "STARTED_AT_UNIX=$STARTED_AT_UNIX"
  echo "STARTED_AT=$STARTED_AT"
)
```

其中 `<snapshot_path>` 来自 v0.2.2 缓存的 snapshot 路径（沿用 connection 对象的 `_snapshot_path`）；`<jobs_dir>` 是 `~/.cache/remote-mcp/jobs`。

**关键不变量**：`echo $PID > pid` 必须出现在三个 `echo XXX=` 之前。这样即使响应丢失，agent 可以通过 `Jobs(name=...)` 读到 pid 文件并恢复管理（started_at 这种非关键信息可以从 SFTP `stat` pid 文件的 mtime 回填，精度足够）。

#### 5.3.4 启动响应解析

成功收到全部三行（`BG_PID=<int>`、`STARTED_AT_UNIX=<int>`、`STARTED_AT=<iso>`）→ 进入 §5.3.5 更新 meta.json。
任一字段缺失或 exec 失败 → 返回（**注意：进程可能仍已启动，因为 pid 文件已写**）：

```
Error: background launch for '<name>' on <host> may have started but
the launch response was lost (timeout or channel error: <detail>).
Check Jobs(name='<name>') — if pid is set, the task is running; clean
up with JobArchive(name='<name>') after the task finishes. If pid is
null, the task did not actually start; remove the panel entry with
JobArchive(name='<name>') (it accepts entries with state='stopped').
```

#### 5.3.5 启动成功后回填 meta.json

`pid` / `started_at` / `started_at_unix` 三个字段从响应解析填入，**通过 SFTP 重新写一次 meta.json**。这次写入失败不撤销已启动的进程；返回 Error 但说明"进程已启动 PID=...，meta 未能更新；agent 可手动 `Bash('echo ... > .../meta.json')` 修复"。这是退化路径，实际很少发生（紧跟在成功 SFTP 路径之后）。

#### 5.3.6 返回（成功）

```
Started background task.
  id: 17
  name: x86_python_build
  log_path: /home/user/tmp/build/log
  pid: 1259443
  started_at: 2026-05-31T08:40:53Z

[host=tjcs_ex_ln3 cwd=/home/user]
```

字段说明：

- `id`：当前 session 内单调递增整数（从 1 开始）
- `name`：与传入或自动生成一致
- `log_path`：与传入一致，或默认 `/tmp/rmcp-bg-<uuid>.log`
- `pid`：远端进程 PID
- `started_at`：远端 ISO-8601 时间戳，UTC（来自 wrap 内 `date -Iseconds -u`）

description 不在返回中（agent 自己传的，无需回显）。

## 6. meta.json schema

每个任务一份，路径 `~/.cache/remote-mcp/jobs/<sid>/<name>/meta.json`，JSON 内容：

```json
{
  "id": 17,
  "name": "x86_python_build",
  "description": "Python 3.13.4 build on x86 login node",
  "command": "bash ~/scripts/login_x86_python.sh",
  "log_path": "/home/user/tmp/build/log",
  "host": "tjcs_ex_ln3",
  "pid": 1259443,
  "started_at": "2026-05-31T08:40:53Z",
  "started_at_unix": 1748685653,
  "kill_requested_at": null,
  "kill_requested_at_unix": null,
  "archived_at": null,
  "archived_at_unix": null
}
```

字段说明：

| 字段 | 类型 | 写入时机 | 说明 |
|---|---|---|---|
| `id` | int | launch 时 | 当前 session 内唯一，单调递增 |
| `name` | str | launch 时 | 用户指定或自动生成 |
| `description` | str | launch 时 | 任务描述（可空） |
| `command` | str | launch 时 | agent 传入的原 string（未展开 wrap） |
| `log_path` | str | launch 时 | 重定向目标路径 |
| `host` | str | launch 时 | 来自 `conn.config.name` |
| `pid` | int \| null | launch 时回填 | 启动响应失败时可能保持 null；`Jobs` 查询时若 meta.json 的 pid 为 null 但 `<jobs_dir>/<sid>/<name>/pid` 文件存在且内容是合法整数，从该文件读取兜底（不写回 meta.json，保持懒加载） |
| `started_at` | str | launch 时 | ISO-8601 UTC，从远端 `date -Iseconds -u` 取得 |
| `started_at_unix` | int | launch 时 | unix 秒，用于 elapsed_sec 计算 |
| `kill_requested_at` | str \| null | JobKill 时 | ISO-8601 UTC |
| `kill_requested_at_unix` | int \| null | JobKill 时 | unix 秒 |
| `archived_at` | str \| null | JobArchive 时 | ISO-8601 UTC |
| `archived_at_unix` | int \| null | JobArchive 时 | unix 秒 |

## 7. 状态机

任务状态完全从 OS 真实情况与 meta.json 推导，**面板不持久化任何"state"字段**（避免与 OS 真实情况冲突造成撒谎）。

| state | 推导条件 |
|---|---|
| `running` | `kill -0 <pid>` 成功 |
| `stopped` | `kill -0 <pid>` 失败 且 `kill_requested_at_unix` 为 null |
| `killed` | `kill -0 <pid>` 失败 且 `kill_requested_at_unix` 非 null |

**边界 1**：`kill -0` 失败 + `kill_requested_at` 非 null + 进程仍活（罕见，kill 请求未生效）→ 因为 `kill -0` 是失败状态，归 `killed`。下次 `kill -0` 成功又会回到 `running`，但实践中这种状态切换不应该长时间存在。

**边界 2**：进程死亡且无 pid 文件（meta.json 的 pid 字段为 null）→ `state=stopped`，pid 字段在返回里保持 null。

**返回值不包含 exit_code**——agent 自己读 `log_path` 判断任务是否正常结束。理由：`exit_code=0` 不能表示"任务逻辑上成功"（如 build 跑完返回 0 但 artifact 缺失也不算成功），引入这个字段反而误导 agent。

**`elapsed_sec` 语义**：始终是"远端当前时间 - started_at_unix"。对 `state=running` 任务即为运行时长；对 `state=stopped`/`killed` 任务**包含**了任务停止后到本次查询为止的时间，并不准确反映实际运行时长。Agent 若需精确运行时长，应在 status.sh 里自行测量并通过 `status_script_output` 返回，或读 `log_path` 文件 mtime。理由：在远端额外记录 stopped_at 增加 wrap 复杂度而收益小（v0.3.0 选择简洁实现，YAGNI）。

## 8. `Jobs` 工具

### 8.1 签名

```python
def jobs_tool(conn, name: str = None, id: int = None,
              filter: str = None) -> str
```

MCP schema：

```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "id": {"type": "integer"},
    "filter": {"type": "string", "enum": ["stopped_unprocessed"]}
  }
}
```

### 8.2 调用模式

| 形态 | 行为 |
|---|---|
| `Jobs()` | 列表所有非归档任务 |
| `Jobs(filter="stopped_unprocessed")` | 列表非归档且 state ∈ {stopped, killed} 的任务 |
| `Jobs(name="X")` 或 `Jobs(id=17)` | 单任务详查（含运行 status.sh 若存在） |

`name` 与 `id` 同时提供 → `Error: provide only one of name or id`。
两者都不提供时按列表模式处理。
filter 与 name/id 同时提供 → filter 忽略（单任务查询不需要过滤），但若被指定的任务实际不存在或已归档，返回 Error。

### 8.3 列表模式返回字段

每条任务一段 JSON 数组项：

```json
[
  {
    "id": 17,
    "name": "x86_python_build",
    "description": "Python 3.13.4 build on x86 login node",
    "host": "tjcs_ex_ln3",
    "pid": 1259443,
    "log_path": "/home/user/tmp/build/log",
    "state": "running",
    "started_at": "2026-05-31T08:40:53Z",
    "elapsed_sec": 14523,
    "kill_requested_at": null
  },
  ...
]
```

外层包一行任务计数：

```
2 active jobs (filter=none):

[{...}, {...}]

[host=tjcs_ex_ln3 cwd=/home/user]
```

如果有任务读取失败：

```
2 active jobs (filter=none); 1 task could not be read:

[{...}, {...}]

errors: [{"name": "broken_one", "reason": "meta.json invalid JSON"}]

[host=tjcs_ex_ln3 cwd=/home/user]
```

注：列表模式下任一关键步骤超时（SFTP listdir / batched cat / batched kill check）→ 抛错中止，**不返回 partial 结果**（详见 §14 C3）。`errors[]` 仅用于单个任务的 meta 解析失败这类与超时无关的局部错误。

### 8.4 单任务模式返回字段

```json
{
  "id": 17,
  "name": "x86_python_build",
  "description": "Python 3.13.4 build on x86 login node",
  "command": "bash ~/scripts/login_x86_python.sh",
  "log_path": "/home/user/tmp/build/log",
  "host": "tjcs_ex_ln3",
  "pid": 1259443,
  "state": "running",
  "started_at": "2026-05-31T08:40:53Z",
  "elapsed_sec": 14523,
  "kill_requested_at": null,
  "status_script_output": {
    "stdout": "...",
    "stderr": "",
    "exit_code": 0,
    "elapsed_sec": 2,
    "error": null
  }
}
```

`status_script_output` 字段：
- 若任务没挂载 status.sh → `null`
- 若挂载了且成功运行 → `{stdout, stderr, exit_code, elapsed_sec, error: null}`
- 若运行超时 → `{stdout: "...", stderr: "...", exit_code: null, elapsed_sec: <timeout>, error: "status script timed out after Xs (SIGHUP sent to script; remote effects may have started)"}`
- 若 SSH 层错误（与脚本无关）→ `{..., error: "exec failed: <reason>"}`

### 8.5 实现操作顺序

**列表模式**：

1. SFTP listdir `~/.cache/remote-mcp/jobs/<sid>/`（忽略 `next_id` 这个非目录文件）
2. 批量 cat 所有 `*/meta.json`：单次 exec `for d in jobs_dir/*/; do echo "===META: $(basename $d)==="; cat $d/meta.json; echo "===END==="; done`，解析输出
3. 过滤 `archived_at_unix != null` 的（除非 filter 要求归档——v0.3.0 无此 filter）
4. 应用 `filter="stopped_unprocessed"`（如有）：先获取 state，过滤
5. 批量获取 state：单次 exec `echo "now=$(date +%s)"; for pid in P1 P2 ...; do kill -0 $pid 2>/dev/null && echo "$pid=A" || echo "$pid=D"; done`，解析
6. 推导每条 state 与 elapsed_sec，组装返回

**单任务模式**：

1. SFTP read `~/.cache/remote-mcp/jobs/<sid>/<name>/meta.json`（若指定 id，先 SFTP listdir 然后扫 meta 找匹配 id）
2. 单次 exec `echo "now=$(date +%s)"; kill -0 <pid> 2>/dev/null && echo A || echo D`
3. 若 status.sh 存在 → 单次 exec 跑脚本（带 timeout 与超时保护，详见 §9.3）
4. 组装返回

### 8.6 路由

`Jobs` 是只读操作 → 走 v0.2.2 的 `_with_retry`（reconnect 后重试一次安全）。

### 9.2 行为

#### 9.2.1 `script != ""` （设置或替换）

1. SFTP write `~/.cache/remote-mcp/jobs/<sid>/<name>/status.sh`（覆盖已有）
2. 立刻跑一次验证：exec `bash --noprofile --norc status.sh` with `timeout`
3. 收集 stdout / stderr / exit_code / elapsed_sec
4. 返回 first-run 详情（**总是接受脚本**，无论 exit_code，详见下文）
5. **例外**：first-run 触发 timeout → SFTP unlink `status.sh`，返回 Error

返回（成功，exit_code=0）：

```

First-run validation:
  exit_code: 0
  elapsed_sec: 1
  stdout: |
    pid=1259443 alive
    progress=26/42
  stderr: (empty)

[host=tjcs_ex_ln3 cwd=/home/user]
```

返回（成功，非零 exit_code——脚本被接受，但 agent 应该看到这是"特殊状态"而不是错误）：

```
Status script attached to 'x86_python_build' (id=17). NOTE: first-run
exited with non-zero code. The script is still attached (non-zero exit
may be intentional, e.g. 'task not yet in expected phase'). Verify the
output below matches your intent; call JobScript again with the same
name to replace.

First-run validation:
  exit_code: 2
  elapsed_sec: 1
  stdout: ...
  stderr: ...

[host=tjcs_ex_ln3 cwd=/home/user]
```

返回（first-run 超时——拒绝）：

```
Error: status script first-run timed out after 30s on tjcs_ex_ln3.
Script has been removed; status script for 'x86_python_build' is now
empty. Likely causes: script logic too slow, or timeout too tight.
Adjust and call JobScript again.

[host=tjcs_ex_ln3 cwd=/home/user]
```

#### 9.2.2 `script == ""`（清除）

1. SFTP stat `status.sh`；不存在则 no-op 成功返回
2. SFTP unlink `status.sh`
3. 不运行验证（无脚本可跑）
4. 返回：

```

[host=tjcs_ex_ln3 cwd=/home/user]
```

### 9.3 脚本运行的隔离与超时

每次 `JobScript` 设置时 + 每次 `Jobs(name=X)` 单任务查询时，**面板内部跑脚本必须**：

1. 用独立的 exec channel
2. 通过 `channel.settimeout` 设置 timeout
3. 超时 → 立即 `channel.close()`（SIGHUP 给脚本进程）
4. 收集已收到的 stdout / stderr 部分
5. 转换为 `status_script_output.error` 字段返回，**不让超时 bubble 整个 Jobs 调用**（§14 C4）

### 9.4 路由

`JobScript` 是状态变更操作 → `NO_RETRY_TOOLS`（与 v0.2.2 Bash / Edit / MultiEdit 同列）。

### 11.2 行为

1. SFTP read meta.json（确认任务存在且未归档）

3. 推导当前 state（`kill -0 <pid>` 一次）
4. state == "running" → `Error: cannot archive running job 'X' (pid=<pid>); kill it first with JobKill(name='X') or wait for it to stop, then archive`
5. SFTP write meta.json with `archived_at`, `archived_at_unix` 字段；**不删除 meta.json 文件、不删除 pid、不删除 status.sh、不删除 log_path**

```
(pid, status.sh, meta.json) remain at ~/.cache/remote-mcp/jobs/<sid>/x86_python_build/
for post-mortem inspection.

[host=tjcs_ex_ln3 cwd=/home/user]
```

### 11.3 同名复用

归档后 `JobArchive` 在 §5.3.1 的"name 唯一性检查"中视为非冲突——agent 可以用同一 name 再次 launch 新任务。新任务会**复用** `<name>/` 目录吗？

**不复用，覆盖**：新 launch 会 SFTP truncate-write 新的 meta.json、pid 文件、（如果 agent 之后 JobScript）status.sh。旧任务的 log_path 文件保持不变（log_path 路径可能与新任务不同；agent 自己负责选择）。**这意味着旧任务的 meta.json 在归档后无法通过 mcp 工具回查**——它被新任务的同名 meta 覆盖了。

替代设计（考虑过未采用）：归档时把整个 `<name>/` 目录改名为 `<name>__<archived_at_unix>/`，新任务用新目录。这样能保留旧 meta.json 可查。**未采用原因**：增加目录改名的 SFTP 操作（rename 在某些 FS 上非原子），且 agent 既然主动归档表示"不再关心这个任务"，post-mortem 主要看的是 log_path 而非 meta.json，meta 的丢失可接受。

如果 agent 想保留旧 meta 可查，建议**不要归档**，只让任务停在 stopped/killed 状态。`Jobs()` 仍会列出。这是工作流上的选择，不依赖工具能力。

### 11.4 路由

`JobArchive` 是状态变更操作 → `NO_RETRY_TOOLS`。

## 12. Session ID（sid）派生

### 12.1 算法

mcp server 进程启动时（`server.py::main`）调用一次：

```python
import os, hashlib, socket
import psutil

def derive_sid():
    try:
        ppid = os.getppid()
        parent = psutil.Process(ppid)
        # create_time 返回浮点 unix 时间（秒，含小数）
        ptime = int(parent.create_time())
        hostname = socket.gethostname()
        raw = f"{ppid}-{ptime}-{hostname}"
        sid = hashlib.sha1(raw.encode()).hexdigest()[:12]
        return sid, "ppid+starttime"
    except Exception as e:
        # 回退到 mcp 进程启动 UUID，写本地文件复用（mcp 进程内）
        import uuid
        sid = uuid.uuid4().hex[:12]
        return sid, f"uuid (psutil fallback: {e})"
```

结果缓存在 `SSHConnection` 或全局变量，整个 mcp 进程生命周期内不变。

### 12.2 派生失败的处理

`psutil` 不可用（极少见）→ 回退路径用 UUID。代价是 mcp server 崩溃重启会变 sid，老 panel 看不到。日志（stderr 与 `RemoteInfo` 工具返回里）必须明确告知 sid 来源是 "ppid+starttime" 还是 "uuid fallback"。

### 12.3 sid 的暴露

`RemoteInfo` 工具的返回里加 `sid` 字段（如 `sid: a1b2c3d4e5f6 (source=ppid+starttime)`），方便 agent 在调试或跨工具复合操作时引用。

### 12.4 跨 session 残留的处理

老 session 的 `~/.cache/remote-mcp/jobs/<old_sid>/` 在新 session 看来是不可见的"垃圾"——v0.3.0 **不**自动清理。用户嫌占空间手动 `rm -rf` 即可。这符合用户在 brainstorm 中明确表示的"不刻意删除 cache"原则。

可选的未来扩展：加一个 `JobsAdminCleanup(older_than_days=N)` 维护工具，由用户主动调用。**v0.3.0 不实现**。

### 12.5 依赖加入

`pyproject.toml` 的 `dependencies` 增加 `psutil>=5.9`。

## 14. 鲁棒性契约（panel 工具必须满足）

继承 v0.2.2 的三条契约（不卡死 / 诚实失败 / 自动恢复），并补充以下 panel 专属约束：

### C1 contextual re-raise

所有面板 op 在抛出底层异常（socket.timeout / paramiko.SSHException / OSError / sftp 异常）前必须 wrap 一层 context 信息：

```python
try:
    sftp.mkdir(jobs_dir)
except (OSError, paramiko.SSHException) as e:
    raise type(e)(f"setting up panel entry for '{name}' on {host}: {e}") from e
```

Agent 看到的 Error 必须明确说出"在哪一步出错"（setting up panel entry / reading panel metadata / writing kill_requested_at / 等等），而非裸 `TimeoutError: timed out`。

### C2 launch 前写 pid 文件再 echo

Wrap 模板中 `echo $PID > pid` 必须出现在 `echo "BG_PID=$PID"` 之前。响应丢失时 agent 通过 `Jobs(name=X)` 可读到 pid 文件。

### C3 `Jobs(list)` first-failure abort

列表模式任一关键步骤（SFTP listdir / batched cat meta / batched kill check）超时或失败 → 抛错中止，**不返回 partial 结果**。理由：50 任务超时通常表示远端 IO 真的卡，下一次 cron 周期重试即可；解析 partial 输出的容错代码增加复杂度而收益小。`errors[]` 字段仅用于单个任务 meta 解析失败（与超时无关）。

### C4 status script 失败不污染整调用

`JobScript` 首次运行 + `Jobs(name=X)` 触发的 status.sh 运行，**必须在面板内部 try/except**，将异常转为 `status_script_output.error` 字段返回，不让其 bubble 到 `_with_retry` / `_with_reconnect_only`。理由：状态脚本是"装饰"，失败时基础字段（state / elapsed_sec / pid）仍应可用。

### C5 `JobKill` 先写 kill_requested_at 再 exec

`JobKill` 必须先成功 SFTP write meta.json 的 kill_requested_at 字段，再 exec kill 命令。理由：保证"kill 命令已发出"这件事在面板状态里诚实可见。若 kill 命令执行失败但 kill_requested_at 已写，下次 `Jobs(name=X)` 会显示 kill_requested_at 非 null + state=running，agent 据此判断"kill 没生效"。

### C6 `JobScript` first-run 超时拒绝并清理

`JobScript` first-run 触发 timeout → SFTP unlink 刚写的 status.sh → 返回 Error 明确告知"脚本已移除"。避免半装状态（脚本在远端但 first-run 没验证过）。

### C7 op_timeout 与 NO_RETRY_TOOLS

| 操作 | timeout 来源 | 默认 | NO_RETRY_TOOLS? |
|---|---|---|---|
| 所有 SFTP（write meta, read meta, mkdir, unlink） | `op_timeout_default` | 60s | Bash / JobKill / JobArchive / JobScript: 是；Jobs: 否 |
| 面板内 batched bash exec（cat meta / kill -0 / date） | `op_timeout_default` | 60s | 同上 |
| status script 运行 | agent 在 `JobScript` 指定 | 无默认（必填） | 同上 |
| `JobKill` 的实际 kill 命令 | 独立短 timeout | 5s | NO_RETRY |

### C8 MCP 进程内不开后台线程

面板状态完全靠"按需查询时去问远端"得到，不维护任何长跑线程、不轮询、不缓存。理由：复杂度爆炸（线程同步、reconnect 期间状态混乱），且 agent 通过 cron 触发 `Jobs(filter=stopped_unprocessed)` 已经覆盖"何时知道任务完成"的需求。

### C9 Worst-case 等待边界

| 操作 | worst-case 等待 |
|---|---|
| `Bash(run_in_background=True, ...)` 启动 | ≤ op_timeout_default（首失败抛错） |
| `Jobs(list)` | ≤ op_timeout_default（C3 first-failure） |
| `Jobs(name=X)` 无状态脚本 | ≤ 2 × op_timeout_default |
| `Jobs(name=X)` 含状态脚本 | ≤ 2 × op_timeout_default + script_timeout |
| `JobKill` | ≤ 2 × op_timeout_default + 5s |
| `JobArchive` | ≤ 3 × op_timeout_default（读 meta + kill -0 检 state + 写 meta） |
| `JobScript` (set) | ≤ op_timeout_default + script_timeout |
| `JobScript` (clear) | ≤ 2 × op_timeout_default |

## 15. 错误消息措辞规范

以下列出本规范引入的所有错误消息原文，实施时必须 word-for-word 一致（agent 可能根据文本做行为决策）：

| 触发场景 | 错误消息 |
|---|---|
| Bash launch 时 name 不合法 | `Error: invalid job name 'X': must match ^[A-Za-z0-9_.-]{1,64}$` |
| Bash launch 时 name 冲突 | `Error: job name 'X' already in active panel; archive the old one with JobArchive(name='X') or pick a different name` |
| Bash launch 时 log_path 父路径冲突 | `Error: log_path parent '/foo/bar' exists but is not a directory; cannot mkdir -p` |
| Bash launch 时 log_path 是目录 | `Error: log_path '/foo/bar' is a directory, not a file` |
| Bash launch 响应丢失 | `Error: background launch for '<name>' on <host> may have started but the launch response was lost (timeout or channel error: <detail>). Check Jobs(name='<name>') — if pid is set, the task is running; clean up with JobArchive(name='<name>') after the task finishes. If pid is null, the task did not actually start; remove the panel entry with JobArchive(name='<name>') (it accepts entries with state='stopped').` |
| Jobs 同时提供 name 与 id | `Error: provide only one of name or id` |
| Jobs / JobKill / JobArchive / JobScript 任务不存在 | `Error: no job with name='X' (or id=N) in current session panel` |
| Jobs / JobKill / JobArchive 任务已归档 | `Error: cannot <op> archived job 'X'`（op = kill / archive / 等） |
| JobArchive 在 running 任务上 | `Error: cannot archive running job 'X' (pid=<pid>); kill it first with JobKill(name='X') or wait for it to stop, then archive` |
| JobKill 命令 5s 超时 | `Error: kill command did not respond within 5s on <host>. kill_requested_at has been recorded; the panel will report state=killed if the process becomes dead. Run Jobs(name='X') to verify current state; if state remains 'running', the kill command may have failed.` |
| JobScript first-run 超时 | `Error: status script first-run timed out after Xs on <host>. Script has been removed; status script for '<name>' is now empty. Likely causes: script logic too slow, or timeout too tight. Adjust and call JobScript again.` |
| 面板 op 触发底层超时（context-wrapped） | `Error: <ExceptionClass>: <op context, e.g. 'setting up panel entry for X on host'>: <original message>` |

## 16. 不在 v0.3.0 范围内

明确写在 spec 中以避免实施时被"补充"进来：

- ❌ 独立的 `RemoteSpawn` 工具——启动入口保持 `Bash(run_in_background=True)`
- ❌ 独立的 `JobStatus` 工具——聚合查询能力以 `JobScript` 形式融入面板
- ❌ `wait` / 阻塞机制——靠 cron 轮询 `Jobs(filter=stopped_unprocessed)`
- ❌ MCP 推送通知（`notifications/*`）——先验证 Claude Code 客户端是否消费再议
- ❌ 独立的 `JobClearScript` 工具——`JobScript(script="")` 即清除
- ❌ `JobSetKillCmd` 持久化 kill 命令——`JobKill` 的 `kill_cmd` 是 just-in-time 参数
- ❌ `JobUnarchive`——归档单向；agent 需重新可见任务只能新 launch
- ❌ status script 的远端路径模式——v0.3.0 仅支持 inline string
- ❌ 自动给同名任务加 uuid 后缀——同名活跃 → Error（agent 必须显式处理）
- ❌ `exit_code` 字段——agent 自己读 log 判断任务成功与否
- ❌ log_tail 默认带在 `Jobs` 返回——agent 用 `Read(log_path)`
- ❌ 跨 host 任务管理——per-host process 模型不变，每个 mcp server 进程只管自己 host 的 sid 子目录
- ❌ 自动清理老 session 的 `<sid>/` 残留——用户手动 `rm -rf`
- ❌ MCP 进程内任何后台线程 / 轮询 / 缓存

## 17. 验收标准（acceptance criteria）

实施完成后，下列场景必须能复现成功：

### A 组：Bash 扩展

- **A1**：`Bash(run_in_background=True, command="sleep 100")` → 返回包含 id / name / log_path / pid / started_at；其中 name 形如 `bg-[a-f0-9]{12}`，log_path 形如 `/tmp/rmcp-bg-[a-f0-9]{12}.log`
- **A2**：`Bash(run_in_background=True, command="sleep 100", name="myjob", log_path="/tmp/myjob.log", description="test")` → 返回里 name=myjob, log_path=/tmp/myjob.log；`Bash("ls /tmp/myjob.log")` 显示文件存在
- **A3**：A2 后再 `Bash(run_in_background=True, command="sleep 1", name="myjob")` → Error: `job name 'myjob' already in active panel...`
- **A4**：`Bash(run_in_background=True, command="sleep 100", log_path="/no/such/parent/log")` → SFTP mkdir 父目录成功（递归创建 `/no/such/parent`），任务正常启动
- **A5**：`Bash(run_in_background=True, command="sleep 100", log_path="/etc/passwd/log")` → Error: `log_path parent '/etc/passwd' exists but is not a directory...`（假设 `/etc/passwd` 是文件）

### B 组：Jobs 查询

- **B1**：`Jobs()` 在没有任何任务时 → `0 active jobs (filter=none):\n\n[]`
- **B2**：A2 之后 `Jobs()` → 1 个任务，state=running，elapsed_sec ≥ 0
- **B3**：A2 任务 sleep 完成后 `Jobs()` → state=stopped
- **B4**：`Jobs(name="myjob")` → 单任务详查，含 command / status_script_output=null
- **B5**：`Jobs(filter="stopped_unprocessed")` 在 B3 之后 → 只列 myjob
- **B6**：`Jobs(name="nonexistent")` → Error: `no job with name='nonexistent'...`
- **B7**：`Jobs(name="X", id=1)` → Error: `provide only one of name or id`

### C 组：JobKill

- **C1**：A2 任务运行中 `JobKill(name="myjob")` → 返回 kill_command="kill -TERM -- -<pid>"; `Jobs(name="myjob")` 显示 kill_requested_at 已设；几秒内 state=killed
- **C2**：归档任务 `JobKill(name="archived")` → Error: `cannot kill archived job...`
- **C3**：`JobKill(name="myjob", kill_cmd="echo would scancel 12345")` → kill_command 字段显示传入命令，状态 kill_requested_at 已设

### D 组：JobArchive

- **D1**：stopped 任务 `JobArchive(name="myjob")` → 返回 archived_at；后续 `Jobs()` 不再列出
- **D2**：running 任务 `JobArchive(name="X")` → Error: `cannot archive running job 'X'...`
- **D3**：归档后 `Bash(run_in_background=True, name="myjob", command="sleep 1")` → 允许（同名复用）
- **D4**：归档后 `Bash("ls ~/.cache/remote-mcp/jobs/<sid>/myjob/")` → meta.json / pid / 等文件仍存在（归档不删文件）

### E 组：JobScript

- **E1**：`JobScript(name="myjob", script="echo hello", timeout=5)` → first-run 验证成功，返回 stdout=hello / exit_code=0
- **E2**：E1 后 `Jobs(name="myjob")` → status_script_output 含 stdout=hello
- **E3**：`JobScript(name="myjob", script="sleep 30", timeout=2)` → Error: `status script first-run timed out after 2s...`; `Jobs(name="myjob")` 之后 status_script_output=null（脚本已被清除）
- **E4**：`JobScript(name="myjob", script="exit 2", timeout=5)` → 接受脚本，返回 first-run exit_code=2 + NOTE 提示
- **E5**：`JobScript(name="myjob", script="", timeout=5)` → 清除脚本；`Jobs(name="myjob")` 显示 status_script_output=null
- **E6**：`JobScript` 缺 timeout → schema validation 失败

### F 组：sid 派生与跨 session 隔离

- **F1**：mcp server 启动后 `RemoteInfo` 返回里包含 `sid: <hex12> (source=ppid+starttime)`
- **F2**：手动 kill 然后 restart mcp server（父进程不变）→ 新进程 RemoteInfo 显示**相同**的 sid；之前的任务在 `Jobs()` 里仍可见
- **F3**：两个独立 mcp server 进程（模拟两个 Claude Code session）连同一 host → 各自看到自己的任务，互不可见

### G 组：鲁棒性（需要 v0.2.3 FlakyTCPProxy 支持）

- **G1**：`Bash(run_in_background=True, ...)` 在 mkdir 阶段网络断 → Error 含 "setting up panel entry for X" context；本地 `~/.cache/remote-mcp/jobs/<sid>/X/` 不残留半成品
- **G2**：`Jobs()` 在 batched cat 阶段超时 → 抛错中止，无 partial 结果
- **G3**：`Jobs(name=X)` 状态脚本运行卡死 → script_timeout 后返回，status_script_output.error 含 "timed out"，state / elapsed_sec 等字段正常
- **G4**：`JobKill` 在写 kill_requested_at 后、exec kill 前网络断 → kill_requested_at 已写；下次 `Jobs(name=X)` 显示 state=running + kill_requested_at 非 null（agent 可知 kill 未生效）

## 18. 实施阶段划分（供 writing-plans 参考）

按底层 → 上层依次实施：

1. **Stage A**：sid 派生（§12）+ `psutil` 依赖 + `RemoteInfo` 暴露 sid
2. **Stage B**：`jobs/` 模块（registry / state / scripts），不接 server，单测覆盖
3. **Stage C**：Bash 扩展（§5）+ 新 BASH_DESC（§13）+ 验收 A 组
4. **Stage D**：`Jobs` 工具（§8）+ 验收 B 组
5. **Stage E**：`JobScript`（§9）/ `JobKill`（§10）/ `JobArchive`（§11）+ 验收 C/D/E 组
6. **Stage F**：跨 session 隔离测试（F 组）+ FlakyTCPProxy 鲁棒性测试（G 组）
7. **Stage G**：文档（docs/reference/tools/ 各工具页 + docs/explanation 加 job-panel 章节 + CLAUDE.md 更新 + CHANGELOG）

每个 stage 内按 TDD 节奏：失败测试 → 实现 → 通过测试 → commit。详细任务分解由 writing-plans 阶段产出。
