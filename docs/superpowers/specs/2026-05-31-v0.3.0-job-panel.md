# remote-mcp v0.3.0 设计规范：后台任务面板

**日期**：2026-05-31
**状态**：待审核 → 待实施
**前身**：增量于 [`2026-05-26-remote-mcp-design.md`](./2026-05-26-remote-mcp-design.md)（v1）、[`2026-05-27-v0.2.0-non-persistent-bash.md`](./2026-05-27-v0.2.0-non-persistent-bash.md)（v0.2.0/v0.2.1）、[`2026-05-28-network-robustness-design.md`](./2026-05-28-network-robustness-design.md)（v0.2.2/v0.2.3）。本规范覆盖 v0.3.0 的"后台任务面板"子系统：扩展 `Bash(run_in_background=True)` 的参数与返回结构，并新增 4 个面板管理工具。其余设计（连接生命周期、SFTP 工具、非持久前台 Bash、cwd 配置、网络鲁棒性契约等）继续以前述 spec 为准。

## 1. 概述与目标

v0.2.x 把 `Bash(run_in_background=True)` 做到了正确——`setsid + nohup + </dev/null + &` 这套防 SIGHUP / 防 stdin 占用 / 防 shell 退出的零件齐全；远端进程在 SSH 断开后仍能存活；pidfile 保证响应丢失时 PID 可找回。但实际使用中，agent 在管理多小时长任务时仍倾向手写 `nohup` 而非使用该工具。原因有两个：

1. **工具描述未告知 agent 实际生成的 shell 命令**。Agent 不知道 `run_in_background=True` 已经包含 setsid + nohup，因此不放心把 18 小时的 build 托付给它，宁可手写 nohup 以"亲眼看到 SIGHUP 防御"。这是文档可发现性问题，而非功能问题。
2. **不支持命名、列表、kill、状态查询等管理操作**。Agent 必须自己用 `pgrep`/`tail`/`kill` 组合每次 30+ 行 boilerplate 才能查询任务状态——量化损失在 18 小时 build 场景下约 75 KB token / session（详见 §3 动机）。

v0.3.0 解决这两个问题：

- **改写 `BASH_DESC`**，把 foreground 与 background 实际生成的 shell 命令展开告知 agent；顺便修正 v0.2.0 残留的"shell state persists across foreground calls"错误描述。
- **扩展 `Bash(run_in_background=True)`** 的参数（`log_path` / `name`）与返回结构（id / name / log_path / pid / started_at）。
- **新增 4 个面板管理工具**：`Jobs`（查询）、`JobKill`（kill）、`JobArchive`（归档）、`JobScript`（挂载状态脚本）。

总工具数从 13 增加到 17。

## 3. 动机

### 3.1 Agent feedback 的两条原始反馈

来自 2026-05-28 同一 session 内 35 秒间隔提交的两条 enhancement feedback（agent 跑 18 小时 LLVM/Python build 后所写），核心数据：

**反馈 #1（聚合状态查询）**：每次任务状态查询需手写 30+ 行 boilerplate：

```bash
pgrep -af login_.*_python.sh 2>&1
echo "===build.log==="
ls -la ~/tmp/python_build/logs/python.build.log 2>&1
echo "===tail build==="
tail -3 ~/tmp/python_build/logs/python.build.log 2>&1
echo "===login.out tail==="
tail -5 ~/tmp/python_build/login.out 2>&1
echo "===install python3?==="
ls ~/install/Python-3.13.4/bin/python3 2>&1 || echo NOT_YET
```

单次返回 1-2 KB（其中大部分是 locale "总用量" 文本、tail 多行原文、pgrep 完整命令行）。一晚 50+ 次 check 累计 ~75 KB token，其中 99% 用来表达"还在跑"这一个 bit 的信息。

**反馈 #2（命名长任务管理）**：长 build 全部用手写 nohup 启动而非 `run_in_background=True`，原因是：

- log 路径不可控（默认 `/tmp/rmcp-bg-<uuid>.log`，不适合 cron 持续监控固定路径）
- 多并发任务用随机 UUID 难以区分（手写时用脚本名 + 自选 log 路径区分）
- 跨 session 续接困难（笔记本休眠醒来后 UUID 丢失）

### 3.2 经过澄清后的真正缺口

与反馈 agent 沟通后澄清：**反馈 #2 三条理由的根本原因是 agent 不知道 `run_in_background=True` 已经包含 setsid + nohup**——agent 误以为该工具仅做"fire-and-forget exec"，因此为了 18 小时长 build 的安全宁可手写 nohup 以"亲眼看到 SIGHUP 防御"。Log 路径、命名、跨 session 续接的痛点在已知 mcp 后台机制完备的前提下，本可以通过工具描述指引 agent 用 `Bash(run_in_background=True, command="bash build.sh > ~/build.log 2>&1")` 加 `pgrep + 已知 log 路径` 的组合解决。

但**反馈 #1 的聚合查询缺口是真实结构性缺失**，无法靠文档解决——agent 没有别的途径在一次 SSH 调用内得到"进程是否还活着 + 进度 + 主日志尾巴 + 失败标记 + 完工 artifact"的结构化聚合。

因此 v0.3.0 的设计基于以下重定位：

- 反馈 #2 的"启动正确性"已由 v0.2.x 解决，只需文档暴露 → §13 `BASH_DESC` 改写
- 反馈 #2 的"log 路径可控、可命名"是 ergonomic 改进 → §5 Bash 新增 `log_path` / `name`
- 反馈 #2 的"跨 session 列表/kill"是真实管理缺口 → §8 / §10 / §11 新增面板工具
- 反馈 #1 的"聚合查询"是真实结构性缺失 → §9 面板支持挂载状态脚本

## 4. 架构总览

面板**元数据**与**状态脚本本体**存储在 mcp server 主机的**本地**文件系统（XDG `~/.local/share/`），由 mcp 进程直接读写——不通过 SSH。这样高延迟远端环境下面板操作不被网络放大成痛苦体验。

远端只保留三类**必须**在远端的文件（统一扁平命名 `remote-mcp-<sid>-<id>-*`）：

- **pid 文件**：远端 bash wrap 自己写，是响应丢失时找回 PID 的兜底
- **status.sh 缓存**：脚本要在远端执行，本地是 source of truth，远端是上传缓存；缓存可被外部清理（agent 手动 `rm` / 系统 cleanup），下次执行时检测到缺失会自动重新上传
- **log 文件**：agent 自己指定路径，或默认 `~/.cache/remote-mcp-<sid>-<id>.log`

```
mcp server 主机（本地）                              远端
~/.local/share/remote-mcp/jobpane/                  ~/.cache/
└── <sid>/                                          ├── remote-mcp-<sid>-<id>-pid          (wrap 写)
    └── <host>/                                     ├── remote-mcp-<sid>-<id>-status.sh    (本地为源，远端缓存)
        ├── next_id                                 └── remote-mcp-<sid>-<id>.log          (默认 log_path 时)
        ├── .id_lock                              
        ├── <id>-meta.json     (活跃任务)         + agent 指定的其他 log_path
        ├── <id>-status.sh     (可选)            
        ├── archive/                              
        │   ├── <id>-meta.json (normal 归档: state ∈ {stopped, killed})
        │   └── <id>-status.sh
        └── zombie/                               
            ├── <id>-meta.json (zombie 归档: kill_failed + as_zombie=True)
            └── <id>-status.sh
```

- `<sid>`：mcp server 进程启动时派生于 PPID + 父进程 start_time（详见 §12），同一 Claude Code session 内 mcp 重启 sid 不变
- `<host>`：mcp server 的 `--host <name>`。同一 session 内连不同 host 的多个 mcp 进程互不干扰
- `<id>`：session+host 范围内单调递增整数。**每任务一个 `<id>-meta.json` 文件**（不是子目录），name 是 meta 内的属性。归档时整文件 mv 到 `archive/` 或 `zombie/` 子目录（按归档类型分流），name 释放。新 launch 的同名任务获得新 id，与归档任务的 `<old_id>-meta.json` 自然并存。Zombie 单独成目录是为了 zombie 数阈值统计（§11.3）只需 listdir 一次，不必扫描全 archive 再过滤 `zombie` 字段
- 远端文件名扁平：`remote-mcp-<sid>-<id>-pid` / `remote-mcp-<sid>-<id>-status.sh` / `remote-mcp-<sid>-<id>.log`——不嵌套；agent 用 `Bash("ls ~/.cache/remote-mcp-*-pid")` 兜底也直观

```
Claude Code
     │ stdio MCP
     ▼
remote-mcp (local Python, --host <name>)
 ├── server.py: list_tools / call_tool / dispatch / 后缀 / WARNING
 │              + 启动时调 jobs.init() 初始化本地目录与 next_id
 ├── connection.py: SSHConnection (paramiko, compress=True, snapshot)
 │                  + exec_with_snapshot(conn, command, timeout) -> ExecResult
 ├── jobs/: 新增模块
 │    ├── sid.py: 派生 session id（PPID + parent start_time）
 │    ├── paths.py: 本地路径计算 + 远端扁平路径计算
 │    ├── init.py: 启动初始化（mkdir / touch next_id）
 │    ├── meta.py: <id>-meta.json 本地读写、id 分配（fcntl flock）、state 缓存写回
 │    ├── state.py: 远端 kill -0 + date 批量查询、4 态状态机推导
 │    ├── scripts.py: status.sh 本地写 + SFTP 上传 + first-run 跑 + 超时
 │    └── constants.py: KILL_FAIL_PER_TASK_THRESHOLD=3、STUCK_KILL_WARN_THRESHOLD=5、ZOMBIE_WARN_THRESHOLD=5
 └── tools/
      ├── bash.py: 扩展 run_in_background 路径；前台路径改为调 exec_with_snapshot
      └── jobs.py / job_kill.py / job_archive.py / job_script.py: 新增
                │ SSH (paramiko Transport, compress=True)
                ▼
              remote host
```

**关键边界**：

- 面板**元数据**与 **status.sh 源文件**完全在 mcp 主机本地。远端仅作 pid / log / status.sh 缓存。
- mcp server 进程内**不开后台线程**——所有面板操作都是同步、有界时间的工具调用。
- mcp server 进程内**不缓存元数据**——每次工具调用按需从本地文件读。本地 IO 廉价。
- sid 派生于父进程 PPID + 父进程 start_time + hostname，保证：
  - 同一 Claude Code session 内 mcp server 崩溃重启 → sid 不变 → 面板保留
  - 不同 Claude Code session 并发 → 不同 sid → 不同子目录 → 互不干扰
  - **已知限制**：用户关闭 Claude Code 重开（含 `claude -c <id>` resume）→ 新 CC 进程，新 PPID → 新 sid → 老 panel 在新 sid 下不可见（任务仍跑在远端，agent 用 `Bash("ls ~/.cache/remote-mcp-*-pid")` 枚举老 sid 兜底，详见 §12.5）

### 4.1 启动初始化

mcp server 进程启动时（`server.py::main`）按顺序执行：

1. 派生 sid（§12.1），缓存到全局
2. 本地 `mkdir -p ~/.local/share/remote-mcp/jobpane/<sid>/<host>/`
3. 本地 `mkdir -p ~/.local/share/remote-mcp/jobpane/<sid>/<host>/archive/`
4. 本地 `mkdir -p ~/.local/share/remote-mcp/jobpane/<sid>/<host>/zombie/`
5. 若 `<sid>/<host>/next_id` 不存在 → 写入 `"0"`
6. 若 `<sid>/<host>/.id_lock` 不存在 → 创建空文件（fcntl 锁需要的载体）

任一步失败 → **mcp 启动直接终止**，打印明确错误（如 `Error: cannot init job panel at ~/.local/share/remote-mcp/jobpane/<sid>/<host>/: <reason>. Check filesystem permissions / disk space.`），由用户修复后重启。**不降级运行**——半工作状态比直接挂掉更糟。

启动初始化只跑一次；后续工具调用假定上述目录与文件已存在，不再重复 mkdir / touch。

## 5. Bash 工具扩展（`run_in_background=True` 路径）

### 5.1 Python 签名

```python
def bash(conn, command: str, *,
         run_in_background: bool = False,
         timeout: float = None,
         description: str = "",
         log_path: str = None,
         name: str = None) -> str:
```

所有新参数（`log_path` / `name`）以及现有可选参数全部 **keyword-only**（`*` 之后），调用方必须用关键字传参。Foreground 路径（`run_in_background=False`）忽略 `log_path` 与 `name`；description 在两条路径下都是 CC native 兼容字段，foreground 时保持 v0.2.x 的"占位未使用"语义，background 时作为面板 description 持久化到 meta.json。

### 5.2 MCP schema（`schemas.py::BASH_SCHEMA`）

```json
{
  "type": "object",
  "properties": {
    "command": {"type": "string"},
    "description": {
      "type": "string",
      "default": "",
      "description": "Brief description of the command (CC native compat). When run_in_background=true, this is also stored as the task description in the panel (max 500 chars; longer is truncated with a notice in the return)."
    },
    "timeout": {"type": "number", "default": 120},
    "run_in_background": {"type": "boolean", "default": false},
    "log_path": {
      "type": "string",
      "description": "Optional. Absolute remote path for stdout+stderr redirection (background only). ### 5.3 Background 启动语义
```

#### 5.3.1 参数校验（启动前同步、纯本地）

| 检查 | 失败行为 |
|---|---|
| `name` 提供时正则 `^[A-Za-z0-9_.-]{1,64}$` | `Error: invalid job name 'X': must match ^[A-Za-z0-9_.-]{1,64}$` |
| `description` 长度 ≤ 500 字符 | 超长截断到 500 字符，返回里附加 `[description truncated to 500 chars]` 提示 |
| `name` 与活跃任务冲突（活跃 = 非归档；本地扫 `<sid>/<host>/*-meta.json`，不含 `archive/`） | `Error: job name 'X' already in active panel; archive the old one with JobArchive(name='X') or pick a different name` |

`name` 未提供时自动生成 `bg-<uuid12>`（如 `bg-a1b2c3d4e5f6`），12 字符 hex 在 session+host 范围内冲突概率可忽略。

`log_path` 父路径合法性在 §5.3.2 步骤 5 通过远端 mkdir 探测（错误同步报告）。

#### 5.3.2 启动顺序

启动初始化（§4.1）已完成，所以 `<sid>/<host>/` 与 `archive/` / `zombie/` / `next_id` / `.id_lock` 都已就绪。每次 launch 只做：

1. **本地原子分配 id**：`flock <sid>/<host>/.id_lock` 内 read+1+write `next_id`，取得 `<id>`
2. **本地校验 name 唯一性**（§5.3.1 已在前置校验描述）；若冲突 → Error
3. **本地 write** `<sid>/<host>/<id>-meta.json`（schema 详见 §6；pid / started_at 字段为 null 等 exec 后回填；**`state` 直接置为 `"running"` + `state_at_unix` = mcp 本地当前 unix 时间**——agent 调 `Bash(run_in_background=True)` 即视为任务启动尝试，state 缓存先记 running；若 launch 实际失败，agent 后续 `Jobs` 调用会刷新到真实 state）
4. **决定 log_path**：参数提供则用之；未提供 → 默认 `~/.cache/remote-mcp-<sid>-<id>.log`（路径里 `~` 由远端 bash 在 wrap 展开；写回 meta.json 与返回给 agent 时统一展开为绝对路径，因 Read 等工具拒绝 `~`）
5. **远端 op #1**：一律执行 `mkdir -p $(dirname <log_path>)`（单次 exec，30s timeout，幂等）。默认 log_path 的父目录 `~/.cache/` 通常已存在；自定义 log_path 父目录可能不存在；统一兜底比加条件判断简单。
   - exit_code = 0 → 继续
   - exit_code != 0 → 解析 stderr：若 "not a directory" → `Error: log_path parent ... exists but is not a directory; cannot mkdir -p`；其他 → `Error: cannot create log_path parent: <stderr>`；同时本地 unlink 步骤 3 写的 `<id>-meta.json`（清理）
6. **远端 op #2**：exec wrap（§5.3.3），等待 BG_PID / STARTED_AT / STARTED_AT_UNIX 三行响应
7. **本地 update** `<id>-meta.json`：填入 pid / started_at / started_at_unix（解析响应）

任一步失败时的清理：仅当步骤 3 已写本地 `<id>-meta.json` 且后续步骤失败时，**本地** unlink 该文件。本地清理同步且通常不会失败；失败也仅在 Error 文本附加 `(partial local file may remain at ~/.local/share/remote-mcp/jobpane/<sid>/<host>/<id>-meta.json)`。无远端文件残留——远端只会在步骤 6 的 wrap exec 之后才产生文件，那时本地状态已就绪。

#### 5.3.3 Wrap 模板（步骤 6 的 exec 内容）

```bash
( setsid nohup bash --noprofile --norc -c '
    source <snapshot_path> 2>/dev/null || true
    <user command>
  ' > <log_path> 2>&1 </dev/null &
  PID=$!
  echo $PID > ~/.cache/remote-mcp-<sid>-<id>-pid
  STARTED_AT_UNIX=$(date +%s)
  STARTED_AT=$(date -u -Iseconds)
  echo "BG_PID=$PID"
  echo "STARTED_AT_UNIX=$STARTED_AT_UNIX"
  echo "STARTED_AT=$STARTED_AT"
)
```

其中 `<snapshot_path>` 来自 v0.2.2 缓存的 snapshot 路径；`<sid>` 和 `<id>` 在 launch 时已知，直接展开。

**关键不变量**：`echo $PID > ~/.cache/remote-mcp-<sid>-<id>-pid` 必须出现在三个 `echo XXX=` 之前。这样即使响应丢失（exec 通道断/超时），远端 pid 文件已落地——agent 通过 `Jobs(id=N)` 触发兜底读取该文件即可恢复管理（详见 §6 的 pid 字段说明）。

#### 5.3.4 启动响应解析（含同步 SFTP 兜底）

**核心契约**：launch 必须**同步确认 pid** 才让任务进入面板；无法确认 pid 的任务**绝不**留在面板里。这样面板里每条任务都对应一个真实存在过的远端进程，避免幽灵任务污染查询。

解析流程：

**A. 正常路径**：exec 完整收到全部三行（`BG_PID=<int>`、`STARTED_AT_UNIX=<int>`、`STARTED_AT=<iso>`）→ 进入 §5.3.5 用响应数据回填 meta。

**B. 响应丢失路径**：exec 失败或任一字段缺失 → **立即同步 SFTP 兜底读** `~/.cache/remote-mcp-<sid>-<id>-pid`（op_timeout_default，60s）：

- **B1. 兜底成功**（文件存在且内容是合法整数）→ 取得 pid；用 SFTP stat 拿 pid 文件 mtime 作为 started_at 的近似值（精度足够）；进入 §5.3.5 用兜底数据回填 meta；返回成功，附带 NOTE 说明"started_at 是从 pid 文件 mtime 推算的，可能比真实启动晚 1-2 秒"
- **B2. 兜底失败**（文件不存在 / SFTP 网络故障 / 内容非整数）→ **本地 unlink `<id>-meta.json`**（任务不入面板）→ 返回 Error：

```
Error: background launch for '<name>' (id=<id>) on <host> could not be
confirmed. exec response was lost (<exec_detail>) AND SFTP fallback
fetch of pid file failed (<sftp_detail>). The task has NOT been added
to the panel.

What this means:
- If SFTP failed because the file is genuinely missing: the wrap never
  ran (or never reached the pid-write step). Nothing to clean up.
- If SFTP failed because of network/timeout: the wrap MAY have started
  and a remote process MAY be orphaned. Recover with:
    Bash("test -f ~/.cache/remote-mcp-<sid>-<id>-pid && cat ~/.cache/remote-mcp-<sid>-<id>-pid")
  If that returns a pid, the orphan exists; kill it with
    Bash("kill -TERM -- -<pid>")
  Then optionally Bash("rm ~/.cache/remote-mcp-<sid>-<id>-pid") to
  clean up the file.

You may safely retry Bash(run_in_background=True, ...) with the same
name (the failed entry has been removed from the panel).
```

#### 5.3.5 启动成功后回填 meta.json

`pid` / `started_at` / `started_at_unix` 三个字段写入本地 meta.json。数据来源：

- 路径 A（响应正常）：来自三行 echo 解析
- 路径 B1（响应丢失但 SFTP 兜底成功）：pid 来自远端 pid 文件内容；started_at_unix 来自远端 pid 文件 stat 的 mtime；started_at 是 mtime 转 ISO

本地 rewrite meta.json。这是本地文件写，几乎不会失败；万一失败（本地磁盘满 / 权限 ROM）→ Error 但说明"进程已启动 PID=...，本地 meta 未能更新；agent 可手动编辑 `~/.local/share/remote-mcp/jobpane/<sid>/<host>/<id>-meta.json` 修复"。

注意：`state` 字段在步骤 3 已经被预置为 `"running"`，§5.3.5 的回填仅覆盖 pid / started_at / started_at_unix，**不改动 state**。任务被加入面板的前提就是 pid 已确认（路径 A 或 B1），所以 state="running" 是诚实的描述。Agent 之后 `Jobs` 调用按真实观察刷新。

#### 5.3.6 返回（成功）

```
Started background task.
  id: 17
  name: x86_python_build
  log_path: /home/user/.cache/remote-mcp-a1b2c3d4e5f6-17.log
  pid: 1259443
  started_at: 2026-05-31T08:40:53Z

[host=tjcs_ex_ln3 cwd=/home/user]
```

字段说明：

- `id`：当前 session+host 范围内单调递增整数（从 1 开始）
- `name`：与传入或自动生成一致
- `log_path`：与传入一致，或默认 `~/.cache/remote-mcp-<sid>-<id>.log`
- `pid`：远端进程 PID
- `started_at`：远端 ISO-8601 时间戳，UTC（来自 wrap 内 `date -u -Iseconds`）

description 不在返回中（agent 自己传的，无需回显）。

| 字段 | 类型 | 写入时机 | 说明 |

## 7. 状态机（4 态）

任务真实状态由"`kill -0 <pid>` 是否成功"与"`kill_requested_at_unix` 是否非 null"决定：

| state | 推导条件 |
|---|---|
| `running` | `kill -0 <pid>` 成功，且 `kill_requested_at_unix` 为 null |
| `stopped` | `kill -0 <pid>` 失败，且 `kill_requested_at_unix` 为 null |
| `killed` | `kill -0 <pid>` 失败，且 `kill_requested_at_unix` 非 null |
| `kill_failed` | `kill -0 <pid>` 成功，且 `kill_requested_at_unix` 非 null（kill 请求已发但进程仍活） |

**衍生标志**（不是新 state，是字段过滤）：

- `stuck_kill`：`state == kill_failed` 且 `kill_attempts.length >= KILL_FAIL_PER_TASK_THRESHOLD`（即 ≥ 3）。
- `zombie`：`archived_at != null` 且 `zombie == true`。

### 7.1 state 缓存、终态不可逆与"陈旧"管理

state 字段持久化在 meta.json：

- **launch 时**预置为 `"running"`（任务刚启动，按定义就是 running；launch 已确认 pid，§5.3.4）
- **Jobs** 与 **JobKill** 在每次成功观察后写回真实 state（可能从 running 更新到 stopped / killed / kill_failed）

`JobArchive` 直接读 meta.json 的 state 缓存，不发起远端观察（详见 §11）。

#### 终态不可逆

`stopped` 和 `killed` 是**终态**——进程死了不能复活，PID 一旦失效不会回到 alive。一旦观察到这两个状态之一并写回 meta，**后续 Jobs 与 JobKill 都不需要再对这个任务发起 `kill -0`**（仍要跑 status.sh 如有挂载，详见 §8.5）。这是一个不影响正确性的优化：

- 列表模式：构建 batched `kill -0` 的 pid 列表时，**只包含 state ∈ {running, kill_failed} 的任务**；终态任务直接复用 meta 缓存值
- 单任务模式：state ∈ {stopped, killed} 时**跳过 kill -0 远端 op**；status.sh（若挂载）**仍正常执行**，因为 agent 可能想用脚本看任务的最终产出

附带正确性收益：避免 PID 复用——远端 OS 在原任务死后把 PID 给别的进程，`kill -0` 又会返回 alive。终态不再观察 = 不会被这条 race 骗。

注意 `kill_failed` **不是终态**——进程仍活，下次 kill 可能生效（→ killed）或自然死亡也归 killed（因为 kill_requested_at 非 null）。所以 kill_failed 仍要持续观察。

#### Archive 的语义：是"已处理完毕的归档"，不是"防陈旧的兜底检查"

如果 agent 调 `JobArchive(name=X)` 而 meta 的 state 还是 `running` 或 `kill_failed`，Archive 拒绝。这**不是**"防止你误归档活着的任务"——是更深的语义：

- agent 上次 Jobs 看到的 state 决定了 agent 心里对这个任务的认知
- 如果 agent 心里还认为它在跑，那 agent 显然还没去查任务是不是完成、没去读 log 看结果
- 在没处理任务结果的情况下就 Archive，本身就是错误的操作

正确的 agent 流程：

```
Bash(run_in_background=True, name="X", ...)   # state="running" (预置)
... 一段时间后 ...
Jobs(name="X")                                # 远端查，刷新 state
# 看到 X 是 stopped
Read("/path/to/X.log")                        # 看任务结果
# 确认结果可接受
JobArchive(name="X")                          # 归档
```

如果 Jobs 看到 state 仍是 running，agent 知道任务还在跑——继续等，别 Archive。如果看到 stopped/killed，先看结果再 Archive。这套流程在缓存与远端真实一致时和不一致时都对：

- 一致：agent 看到的就是真相
- 不一致（缓存陈旧）：agent 也只在缓存说 stopped/killed 时才动手——而那个状态是不可逆终态，缓存说 stopped 远端就一定 stopped

**陈旧 state 的影响**（按场景）：

| 操作 | 影响 |
|---|---|
| `Bash` launch 的 name 冲突检查 | 用"meta 是否在 active 区"判定活跃，与 state 无关；不受影响 |
| `JobArchive`（默认） | state == running / kill_failed 时拒绝，提示 agent 先 Jobs 刷新并查看结果；这是正确语义，不是 bug |
| `JobArchive(as_zombie=True)` | 要求 `state == kill_failed`；同上 |
| `Jobs` 自身 | 这次调用会更新缓存；不存在陈旧问题 |
| `JobKill` | 这次调用会通过 packed exec 拿到最新 alive 状态并写回；不存在陈旧问题 |

### 7.2 边界

**边界 1**：`kill -0` 失败 + 进程仍活（理论可能：UID 不同 / 极端 race）。同 UID 下不会出现；v0.x 设计前提是 mcp 与任务同 UID。

**边界 2**：进程死亡且 pid 字段为 null（启动响应丢失 + 远端 pid 文件也读不到）→ 推导为 stopped，pid 在返回里保持 null。

### 7.3 返回值不包含 exit_code

Agent 自己读 `log_path` 判断任务是否正常结束。理由：`exit_code=0` 不能表示"任务逻辑上成功"（如 build 跑完返回 0 但 artifact 缺失也不算成功），引入这个字段反而误导 agent。

### 7.4 `elapsed_sec` 语义

始终是"远端当前时间 - started_at_unix"。对 `state=running` 任务即为运行时长；对 `state=stopped`/`killed`/`kill_failed` 任务**包含**了任务停止后到本次查询为止的时间，并不准确反映实际运行时长。Agent 若需精确运行时长，应在 status.sh 里自行测量并通过 `status_script_output` 返回，或读 `log_path` 文件 mtime。理由：在远端额外记录 stopped_at 增加 wrap 复杂度而收益小（v0.3.0 选择简洁实现，YAGNI）。

### 8.5 实现操作顺序（含 state 缓存写回与终态优化）

**列表模式**：

1. **本地** listdir 决定扫描区：`<sid>/<host>/`（默认 / stopped_unprocessed / stuck_kill）或 `<sid>/<host>/zombie/`（zombies）。忽略 `next_id` / `.id_lock` / `<id>-status.sh` 等非 meta 文件，且**不**进入 `archive/` 与 `zombie/` 子目录
2. **本地** 读所有 `<id>-meta.json` 反序列化
3. 按 filter 应用纯 meta 字段过滤（zombie 字段、kill_attempts.length 等）
4. **过滤待观察的 pid 列表**：只收集 state ∈ {running, kill_failed} 的任务 pid；state ∈ {stopped, killed} 的任务跳过（终态不可逆，§7.1）
5. **远端 op #1（条件，仅当待观察 pid 列表非空）**：单次 batched exec `echo "now=$(date +%s)"; for pid in P1 P2 ...; do kill -0 $pid 2>/dev/null && echo "$pid=A" || echo "$pid=D"; done`（用 `exec_with_snapshot` 跑，§19）。如果待观察列表为空（全是终态），跳过这次 exec，用本地时钟近似 now（误差几秒，对 elapsed_sec 显示无关紧要）
6. 对**被观察的任务**推导 state（结合 ALIVE 与 kill_requested_at），按 state 二次过滤；终态任务直接复用 meta 缓存值
7. **本地** 把新推导出的 state 与当前 unix 时间写回**被观察任务**的 meta.json（终态任务的 meta 不重写，无需要）
8. 组装返回

**单任务模式**：

1. **本地** 解析定位：`id=N` → 直接 `<sid>/<host>/<id>-meta.json`，失败则依次尝试 `archive/<id>-meta.json`、`zombie/<id>-meta.json`；`name=X` → 扫 active 区找匹配，失败则扫 archive 区，再失败扫 zombie 区
2. **本地** read meta；pid 字段已在 launch 时被确认（§5.3.4），永远非 null。如遇极端情况 meta 被人手动改坏导致 pid=null → 返回 Error `task '<X>' meta is corrupted (pid missing); investigate ~/.local/share/remote-mcp/jobpane/<sid>/<host>/<id>-meta.json manually`
3. **kill -0 观察（条件）**：若 state ∈ {running, kill_failed} → **远端 op #1**：单次 exec `echo "now=$(date +%s)"; kill -0 <pid> 2>/dev/null && echo A || echo D`，推导 state，本地写回 state + state_at_unix。若 state ∈ {stopped, killed} → **跳过 kill -0**，直接用 meta 缓存值（含 state、state_at_unix）
4. **status 脚本运行（条件，无论 state 是否终态都执行）**：若 meta 的 `script_timeout` 非 null → **远端 op #2**：先 SFTP stat `~/.cache/remote-mcp-<sid>-<id>-status.sh`，缺失则从本地 `<id>-status.sh` 上传一份；再用 `exec_with_snapshot` 跑该脚本，timeout 用 `script_timeout`。终态任务也要跑——agent 可能用脚本看任务的最终产出（log 解析、artifact 检查等）
5. 组装返回

**远端 op 数总结**（单任务模式）：

| 场景 | kill -0 | status.sh stat | status.sh 上传 | status.sh 执行 | 合计 |
|---|---|---|---|---|---|
| 终态 + 无 status.sh | 0 | 0 | 0 | 0 | **0** |
| 终态 + 有 status.sh（缓存命中） | 0 | 1 | 0 | 1 | **2** |
| 终态 + 有 status.sh（缓存缺失） | 0 | 1 | 1 | 1 | **3** |
| 非终态 + 无 status.sh | 1 | 0 | 0 | 0 | **1** |
| 非终态 + 有 status.sh（缓存命中） | 1 | 1 | 0 | 1 | **3** |
| 非终态 + 有 status.sh（缓存缺失） | 1 | 1 | 1 | 1 | **4** |

列表模式远端 op 数 ≤ 1（无论任务数）；全终态时 0。

### 8.6 路由

`Jobs` 在 v0.3.0 多出"写回 state 到本地 meta"的副作用，但写入是幂等的（同样 state + 同样 unix 时间），且仅修改本地缓存，对远端无副作用。重试 Jobs 安全 → 仍保留在 v0.2.2 的 `_with_retry` 白名单。

## 10. `JobKill` 工具

### 10.1 签名

```python
def job_kill_tool(conn, name: str = None, id: int = None,
                  kill_cmd: str = None) -> str
```

MCP schema：

```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "id": {"type": "integer"},
    "kill_cmd": {
      "type": "string",
      "description": "Optional. Full shell command to execute on remote to kill the task. Defaults to 'kill -TERM -- -<pid>' (negates pid to signal the process group; works because launch used setsid). Use this for Slurm: kill_cmd='scancel 12345'; for SIGKILL: kill_cmd='kill -KILL -- -<pid>'; for runtime-specific shutdown: kill_cmd='kill -USR1 <pid>'."
    }
  }
}
```

### 10.2 行为

**JobKill 是纯粹的"发一次 kill"操作**——不做自动 zombify。Zombify 由 agent 在判断重试无望后通过 `JobArchive(name=X, as_zombie=True)` 手动触发（§11）。

操作步骤：

1. **本地** read meta（确认任务存在且未归档；已归档 → Error）
2. **本地** write meta with `kill_requested_at` / `kill_requested_at_unix` 更新为当前时间（**必须先成功写入才进入第 3 步**，§14 C5）
3. **远端 op #1**：用 `exec_with_snapshot` 跑 packed command（5 秒短 timeout），把 kill 命令执行与紧随的存活验证打包在一次 exec 里：

   ```bash
   <kill_cmd>
   KE=$?
   sleep 0.1
   kill -0 <pid> 2>/dev/null
   AE=$?
   echo "KILL_EXIT=$KE"
   echo "ALIVE_EXIT=$AE"
   ```

   解析输出得到 `kill_cmd` 的 exit_code 与本次 kill 后进程是否仍活（`ALIVE_EXIT=0` 即仍活）。`sleep 0.1` 给信号传递留个窗口（多数情况 SIGTERM 立即生效；少数被 handler 拦截的进程需要这点时间反应）。
4. **本地** append 这次尝试到 `kill_attempts` 数组：`{at, at_unix, kill_cmd, exit_code: KE, stdout, stderr}`
5. **本地** 推导 state：`ALIVE_EXIT=0` + `kill_requested_at` 非 null → state=kill_failed；`ALIVE_EXIT=1` + `kill_requested_at` 非 null → state=killed。写回 meta 的 state + state_at_unix
6. 组装返回（含本次 attempt 详情 + 如 `ALIVE_EXIT=0` 触发 L1/L2 双层告警判定，详见 §10.3）

### 10.3 双层告警

#### 10.3.1 L1：单任务告警

触发条件：本次 attempt 写入后 `kill_attempts.length >= KILL_FAIL_PER_TASK_THRESHOLD`（默认 3） AND §10.2 步骤 3 中 packed exec 返回 `ALIVE_EXIT=0`（进程仍活，即 state 此刻就是 kill_failed）。

L1 内容追加在返回末尾：

```
NOTE: this task has 3 failed kill attempts now. If the process resists
further signals, try `kill -KILL -- -<pid>`, `scancel --signal=KILL`,
or runtime-specific shutdown commands. After exhausting retries, give
up via JobArchive(name='X', as_zombie=True) to move it to the zombie
queue (it will keep running on remote unmanaged).
```

注：L1 用本次 kill 之后立刻拿到的 `ALIVE_EXIT` 来判断，不需要再发额外一次 `kill -0`。如果 kill 生效（`ALIVE_EXIT=1`），state 此刻为 killed，**不触发 L1**。

#### 10.3.2 L2：host 聚合告警

触发条件：L1 已触发 AND **本 host 上**符合"stuck_kill"条件的任务数（state == kill_failed AND kill_attempts.length >= 3 AND archived_at == null）`>= STUCK_KILL_WARN_THRESHOLD`（默认 5）。

L2 内容（在 L1 之后追加）：

```
==================================================================
WARNING: <host> has 5 jobs with persistent kill failures.
==================================================================

These are tasks where the panel issued kill but the process keeps
running. Pattern of failure across multiple tasks suggests:
  - your kill_cmd choices may be inappropriate (wrong signal / wrong
    PID / missing setsid in launch); inspect kill_attempts arrays
    via Jobs(name=...) to compare what you tried
  - or the remote host may have signal-resistant processes (D-state
    IO, kernel issues, sudoed root)

Consider pausing automation and investigating before more launches.
List affected tasks: Jobs(filter='stuck_kill').
```

L2 统计通过 listdir + 解析 active 区所有 meta.json 完成，全本地，廉价。

## 11. `JobArchive` 工具

### 11.1 签名

```python
def job_archive_tool(conn, name: str = None, id: int = None,
                     as_zombie: bool = False) -> str
```

MCP schema：

```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "id": {"type": "integer"},
    "as_zombie": {
      "type": "boolean",
      "default": false,
      "description": "Only meaningful when archiving a kill_failed task. Setting True acknowledges 'I give up trying to kill this; the process keeps running on remote but the panel forgets it'. Archived task gets zombie=true and counts toward host's zombie threshold. Default false means archive only stopped/killed (per cached state in meta — refresh via Jobs first if needed); calling on kill_failed without as_zombie returns Error guiding you to retry-or-zombify."
    }
  }
}
```

### 11.2 行为（纯本地，零远端 op）

JobArchive 完全基于 meta.json 的缓存 state，**不发起任何远端调用**。

**何时调 Archive**：Archive 的语义是"对已结束（state 为 stopped/killed）并已处理完结果的任务做归档"，不是"把还在跑或未确认的任务从面板里踢出去"。典型流程见 §7.1 末尾的工作流示例：launch → Jobs 刷新 → 读 log 看结果 → Archive。

按 meta 中 state 字段分支（state 永远非 null，§7.1）：

| state | `JobArchive(name=X)`（默认） | `JobArchive(name=X, as_zombie=True)` |
|---|---|---|
| `running` | `Error: task 'X' is in state 'running' per panel (last observed at <ts>). Archive is for tasks you have processed the results of. If you believe the task may have finished: call Jobs(name='X') to refresh state, then Read(log_path) to review results, then archive.` | `Error: as_zombie=True requires state=kill_failed; this task is 'running' — use plain JobArchive(name='X')` |
| `stopped` | ✓ 接受，归 `archive/` | `Error: as_zombie=True requires state=kill_failed; this task is 'stopped' — use plain JobArchive(name='X')` |
| `killed` | ✓ 接受，归 `archive/` | `Error: as_zombie=True requires state=kill_failed; this task is 'killed' — use plain JobArchive(name='X')` |
| `kill_failed` | `Error: cannot archive task 'X' in state 'kill_failed' (pid=<pid>, kill_attempts=N). Either: call JobKill(name='X') again to retry (try kill_cmd='kill -KILL -- -<pid>' or runtime-specific commands), or give up via JobArchive(name='X', as_zombie=True) to move it to the zombie queue.` | ✓ 接受，归 `zombie/` + 标 `zombie=true`；可能触发 escalation warning |

接受路径的操作步骤：

1. **本地** rewrite `<id>-meta.json` 添加 `archived_at` / `archived_at_unix`（若 as_zombie=True，再加 `zombie=true`）
2. **本地** `mv <id>-meta.json` → 目标目录（默认 archive/，as_zombie=True 时 zombie/）下的 `<id>-meta.json`
3. **本地** 若 `<id>-status.sh` 存在 → `mv` 到同一目标目录下的 `<id>-status.sh`
4. 若 as_zombie=True，**本地** `len(os.listdir(zombie/))` 统计 zombie 数（直接看目录文件数，不需要解析 meta）；若 `>= ZOMBIE_WARN_THRESHOLD`（默认 5），返回末尾追加 escalation warning（§11.3）
5. 组装返回

**远端文件原样不动**：pid 文件保留（agent 可继续 `cat ~/.cache/remote-mcp-<sid>-<id>-pid` 兜底）、status.sh 缓存保留（不再被引用，下次 launch 也不会复用同 id）、log 文件保留（agent 仍可读历史输出）。

### 11.3 Escalation warning（zombie 数 ≥ 5）

```
==================================================================
ESCALATION WARNING: zombie count on <host> is now 5 (>= threshold).
==================================================================

Possible causes (review BEFORE assuming remote server is broken):

1. Recent zombie tasks may share a root cause — inspect their
   attempt histories with Jobs(filter='zombies') and Jobs(id=N) for
   each. If they all failed the same kill command, the issue may be
   in your kill_cmd choice (e.g. wrong PID, missing setsid, wrong
   signal for the runtime).
2. If kill_cmd exit codes were all 0 but processes still alive, the
   remote process is genuinely refusing to die — possibly stuck in
   uninterruptible IO (D state), kernel bug, or sudoed root process
   you can't signal as your user.
3. Only after ruling out the above: remote server may be unhealthy.

RECOMMENDED: stop the current task loop and SSH into <host> manually
to investigate. Continued operation may produce more zombies.
```

注意：**仅本次 `JobArchive(as_zombie=True)` 调用触发**该 escalation；后续工具调用不重复 prefix warning（避免 noise）。Agent 应当看到一次就行动。

## 12. Session ID（sid）派生

### 12.1 算法

mcp server 进程启动时（`server.py::main`，启动初始化 §4.1 之前）调用一次：

```python
import os, hashlib, socket
import psutil

def derive_sid():
    try:
        ppid = os.getppid()
        parent = psutil.Process(ppid)
        # create_time 返回浮点 unix 时间（秒，含小数）
        ptime = int(parent.create_time())
        hostname = socket.gethostname()
        raw = f"{ppid}-{ptime}-{hostname}"
        sid = hashlib.sha1(raw.encode()).hexdigest()[:12]
        return sid, "ppid+starttime"
    except Exception as e:
        # 回退到 mcp 进程启动 UUID
        import uuid
        sid = uuid.uuid4().hex[:12]
        return sid, f"uuid (psutil fallback: {e})"
```

结果缓存在全局变量或 `SSHConnection`，整个 mcp 进程生命周期内不变。

### 12.2 派生失败的处理

`psutil` 不可用（极少见）→ 回退路径用 UUID。代价是 mcp server 崩溃重启会变 sid，老 panel 看不到。日志（stderr 与 `RemoteInfo` 工具返回里）必须明确告知 sid 来源是 "ppid+starttime" 还是 "uuid fallback"。

### 12.3 sid 的暴露

`RemoteInfo` 工具的返回里加 `sid` 字段（如 `sid: a1b2c3d4e5f6 (source=ppid+starttime)`），方便 agent 在调试或跨工具复合操作时引用。

### 12.4 跨 session 残留

老 session 的 `~/.local/share/remote-mcp/jobpane/<old_sid>/` 在新 session 看来是不可见的"垃圾"——v0.3.0 **不**自动清理。用户嫌占空间手动 `rm -rf` 即可。远端的扁平 `~/.cache/remote-mcp-<old_sid>-*` 同理保留。

可选的未来扩展：加 `JobsAdminCleanup(older_than_days=N)` 维护工具。**v0.3.0 不实现**。

### 12.5 已知限制：Claude Code 重启 / resume

Claude Code 没有把 session id 通过环境变量或 MCP 初始化握手传给子进程（验证截至 2026-05-30；GitHub issue [#25642](https://github.com/anthropics/claude-code/issues/25642) 提议但未实现）。因此当：

- 用户关闭 CC 重开
- 用户执行 `claude -c <session_id>` resume 历史 session
- 用户执行 `claude --continue` resume 最近 session

→ 新 CC 进程的 PPID 与 v0.3.0 当前 mcp 实例都不同 → mcp 启动后派生出**新的** sid → 老 panel 在新 sid 下不可见。

老 session 的任务**不会丢失**——远端进程仍在跑，本地 meta 文件仍在老 `<sid>/` 目录里——只是 panel 工具看不到。Agent 兜底回收办法：

```bash
# 列出本机所有历史 sid 的远端 pid 文件
Bash("ls ~/.cache/remote-mcp-*-pid 2>/dev/null")

# 检查老 pid 是否还活
Bash("for f in ~/.cache/remote-mcp-*-pid; do
        pid=$(cat $f)
        kill -0 $pid 2>/dev/null && echo \"$f: pid=$pid ALIVE\" || echo \"$f: pid=$pid dead\"
      done")

# 老任务的 log 也可读
Read("/home/user/.cache/remote-mcp-<old_sid>-<id>.log")

# 想 kill 直接 Bash
Bash("kill -TERM -- -<pid>")
```

未来 CC 暴露 session id 后，v0.4.0 可以无缝升级（同一 session 不论 mcp 实例几次重启都共享 sid）。

## 14. 鲁棒性契约（panel 工具必须满足）

继承 v0.2.2 的三条契约（不卡死 / 诚实失败 / 自动恢复），并补充以下 panel 专属约束：

### C1 contextual re-raise

所有面板远端 op 在抛出底层异常（socket.timeout / paramiko.SSHException / OSError / sftp 异常）前必须 wrap 一层 context 信息：

```python
try:
    result = exec_with_snapshot(conn, "mkdir -p ...", timeout=...)
except (OSError, paramiko.SSHException) as e:
    raise type(e)(f"creating log_path parent for '{name}' (id={id}) on {host}: {e}") from e
```

Agent 看到的 Error 必须明确说出"在哪一步出错"，而非裸 `TimeoutError: timed out`。

本地 IO 出错（极少）同样 contextual：`raise type(e)(f"reading local panel meta for id={id}: {e}") from e`。

### C2 launch 前写 pid 文件再 echo + 同步兜底

Wrap 模板中 `echo $PID > ~/.cache/remote-mcp-<sid>-<id>-pid` 必须出现在三个 `echo XXX=` 之前。响应丢失时 **launch 自身**同步发起 SFTP 兜底读取该 pid 文件（§5.3.4 路径 B）；任务要么 pid 确认成功入面板，要么彻底失败被清理出本地。**不存在"task in panel but pid is null"的中间态。**

### C3 `Jobs(list)` first-failure abort

列表模式任一关键步骤（本地 listdir / 本地 read meta / 远端 batched kill check）超时或失败 → 抛错中止，**不返回 partial 结果**。理由：50 任务超时通常表示远端 IO 真的卡，下一次 cron 周期重试即可；解析 partial 输出的容错代码增加复杂度而收益小。`errors[]` 字段仅用于单个任务 meta 解析失败（如 meta.json 损坏，本地 IO 与超时无关）。

### C4 status script 失败不污染整调用

`JobScript` 首次运行 + `Jobs(name=X)` 触发的 status.sh 运行，**必须在面板内部 try/except**，将异常转为 `status_script_output.error` 字段返回，不让其 bubble 到 `_with_retry` / `_with_reconnect_only`。理由：状态脚本是"装饰"，失败时基础字段（state / elapsed_sec / pid）仍应可用。

### C5 `JobKill` 先写 kill_requested_at 再 exec

`JobKill` 必须先成功本地 write meta.json 的 kill_requested_at 字段，再 exec kill 命令。理由：保证"kill 命令已发出"这件事在面板状态里诚实可见。若 kill 命令执行失败但 kill_requested_at 已写，下次 `Jobs(name=X)` 会刷新 state 为 kill_failed / running，agent 据此判断"kill 没生效"。

### C6 `JobScript` first-run 超时拒绝并清理

`JobScript` first-run 触发 timeout → 远端 SFTP unlink 远端 status.sh + 本地 unlink `<id>-status.sh` + 本地 meta 把 script_timeout 设回 null → 返回 Error 明确告知"脚本已移除"。避免半装状态。

### C7 op_timeout 与 NO_RETRY_TOOLS

| 操作 | timeout 来源 | 默认 | NO_RETRY_TOOLS? |
|---|---|---|---|
| 本地 IO（read / write / listdir / mkdir / mv / unlink） | 无（本地 fs 同步调用） | n/a | Bash / JobKill / JobArchive / JobScript: 是；Jobs: 否 |
| 远端 SFTP（pid 兜底读、status.sh 写/删） | `op_timeout_default` | 60s | 同上 |
| 远端 exec（`exec_with_snapshot`） | 由调用方传入 | 见下 | 同上 |
| Bash 前台命令 | bash_timeout_default | 120s | NO_RETRY |
| Bash 背景 wrap exec（等 BG_PID 响应） | 独立短 timeout | 10s | NO_RETRY |
| log_path parent mkdir 远端 exec | 独立短 timeout | 30s | NO_RETRY |
| 面板内 batched kill -0 + date | `op_timeout_default` | 60s | 见上 |
| status script 运行 | agent 在 `JobScript` 指定 | 无默认（必填） | 见上 |
| `JobKill` 的实际 kill 命令 | 独立短 timeout | 5s | NO_RETRY |

### C8 MCP 进程内不开后台线程

所有状态按需查询。面板元数据存本地但**也不进进程内缓存**——每次工具调用重新读本地文件（开销 ms 级，无需缓存）。

### C9 三个写死常量

```python
KILL_FAIL_PER_TASK_THRESHOLD = 3   # 单任务 L1 告警阈值
STUCK_KILL_WARN_THRESHOLD    = 5   # host 级 L2 告警阈值
ZOMBIE_WARN_THRESHOLD        = 5   # host 级 zombie escalation 阈值
```

不进 config.yaml，写死在 `remote_mcp/jobs/constants.py`。

### C10 Worst-case 等待边界

| 操作 | worst-case 等待 |
|---|---|
| MCP 启动初始化（§4.1） | 仅本地 mkdir / touch，毫秒级；失败即终止 |
| `Bash(run_in_background=True, ...)` 启动（默认 log_path，响应正常） | ≤ 30s（log_path mkdir） + ≤ 10s（wrap exec） |
| `Bash(run_in_background=True, ...)` 启动（响应丢失走 SFTP 兜底） | + ≤ op_timeout_default（兜底 SFTP read + stat） |
| `Jobs()` 列表（全终态任务） | 本地 only，毫秒级 |
| `Jobs()` 列表（有非终态任务） | ≤ op_timeout_default（C3 first-failure） |
| `Jobs(id=N)` 终态 + 无 status.sh | 本地 only，毫秒级 |
| `Jobs(id=N)` 终态 + 有 status.sh | ≤ op_timeout_default + script_timeout |
| `Jobs(id=N)` 非终态 + 无 status.sh | ≤ op_timeout_default |
| `Jobs(id=N)` 非终态 + 有 status.sh | ≤ 2 × op_timeout_default + script_timeout |
| `JobKill` | ≤ 5s |
| `JobArchive`（任何形态） | 本地 only，毫秒级 |
| `JobScript` (set) | ≤ op_timeout_default + script_timeout |
| `JobScript` (clear) | 本地 only，毫秒级 |

## 16. 不在 v0.3.0 范围内

明确写在 spec 中以避免实施时被"补充"进来：

- ❌ 独立的 `RemoteSpawn` 工具——启动入口保持 `Bash(run_in_background=True)`
- ❌ 独立的 `JobStatus` 工具——聚合查询能力以 `JobScript` 形式融入面板
- ❌ `wait` / 阻塞机制——靠 cron 轮询 `Jobs(filter=stopped_unprocessed)`
- ❌ MCP 推送通知（`notifications/*`）——先验证 Claude Code 客户端是否消费再议
- ❌ 独立的 `JobClearScript` 工具——`JobScript(script="")` 即清除
- ❌ `JobSetKillCmd` 持久化 kill 命令——`JobKill` 的 `kill_cmd` 是 just-in-time 参数
- ❌ `JobUnarchive`——归档单向；agent 需重新可见任务只能新 launch
- ❌ 自动 zombification（按 N 次失败计数自动转 zombie）——agent 通过 `JobArchive(as_zombie=True)` 手动触发，避免自动机制掩盖 bug
- ❌ status script 的远端路径模式——v0.3.0 仅支持 inline string，本地为源 + 远端缓存
- ❌ 自动给同名任务加 uuid 后缀——同名活跃 → Error（agent 必须显式处理）
- ❌ `exit_code` 字段——agent 自己读 log 判断任务成功与否
- ❌ log_tail 默认带在 `Jobs` 返回——agent 用 `Read(log_path)`
- ❌ 跨 host 任务管理——per-host process 模型不变
- ❌ 自动清理老 session 的本地/远端残留——用户手动清理
- ❌ MCP 进程内任何后台线程 / 轮询 / 缓存
- ❌ 阈值常量配置化——全部写死在 `constants.py`，不进 config.yaml
- ❌ 跨 CC 重启 / resume 的 panel 续接——CC 当前不暴露 session id（issue #25642 未实现）；待上游支持
- ❌ `--session-id` CLI flag 让用户手动指定 sid——不暴露此参数（保持默认 PPID 派生语义简单一致）

## 17. 验收标准（acceptance criteria）

实施完成后，下列场景必须能复现成功：

### A 组：Bash 扩展

- **A1**：`Bash(run_in_background=True, command="sleep 100")` → 返回包含 id / name / log_path / pid / started_at；name 形如 `bg-[a-f0-9]{12}`，log_path 形如 `~/.cache/remote-mcp-[a-f0-9]{12}-1.log`
- **A2**：`Bash(run_in_background=True, command="sleep 100", name="myjob", log_path="/tmp/myjob.log", description="test")` → 返回 name=myjob, log_path=/tmp/myjob.log；`Bash("ls /tmp/myjob.log")` 显示文件存在
- **A3**：A2 后再 `Bash(run_in_background=True, command="sleep 1", name="myjob")` → Error: `job name 'myjob' already in active panel...`
- **A4**：`Bash(run_in_background=True, command="sleep 100", log_path="/tmp/no/such/parent/log")` → 远端 mkdir 父目录成功（递归创建 `/tmp/no/such/parent`），任务正常启动
- **A5**：`Bash(run_in_background=True, command="sleep 100", log_path="/etc/passwd/log")` → Error: `log_path parent '/etc/passwd' exists but is not a directory...`（假设 `/etc/passwd` 是文件）
- **A6**：A1 后查本地：`~/.local/share/remote-mcp/jobpane/<sid>/<host>/1-meta.json` 存在；远端 `~/.cache/remote-mcp-<sid>-1-pid` 存在；本地 meta 的 `pid` 字段已填充（**非 null**——launch 路径 A 或 B1 都保证 pid 确认）；`state` 字段为 `"running"`（launch 时预置；agent 后续 Jobs 调用会刷新到真实值）
- **A7**：模拟 wrap echo 响应丢失但远端 pid 文件已写：launch 自动 SFTP 兜底成功 → 任务进入面板，pid 字段从兜底取得，started_at 从 pid 文件 mtime 推算；返回带 NOTE 说明 started_at 是近似值
- **A8**：模拟 wrap echo 响应丢失且 SFTP 兜底失败（pid 文件不存在）：返回 §5.3.4 路径 B2 的 Error；本地 `<id>-meta.json` 已被清理（不存在）；任务不在面板

### B 组：Jobs 查询

- **B1**：`Jobs()` 在没有任何任务时 → `0 active jobs (filter=none):\n\n[]`
- **B2**：A2 之后 `Jobs()` → 1 个任务，state=running，elapsed_sec ≥ 0；本地 meta 的 state 字段已被写为 `"running"`
- **B3**：A2 任务 sleep 完成后 `Jobs()` → state=stopped；meta 的 state 字段更新为 `"stopped"`
- **B4**：`Jobs(name="myjob")` → 单任务详查，含 command / kill_attempts / status_script_output=null
- **B5**：`Jobs(filter="stopped_unprocessed")` 在 B3 之后 → 只列 myjob
- **B6**：`Jobs(name="nonexistent")` → Error: `no job with name='nonexistent'...`
- **B7**：`Jobs(name="X", id=1)` → Error: `provide only one of name or id`
- **B8**：`Jobs(name="X", filter="zombies")` → Error: `filter is for list mode...`

### C 组：JobKill

- **C1**：A2 任务运行中 `JobKill(name="myjob")` → 返回 kill_command="kill -TERM -- -<pid>", state_now=killed；meta 的 state 字段已更新为 killed
- **C2**：归档或 zombie 任务 `JobKill(name="archived")` → Error: `cannot kill job 'archived' — it is in archive or zombie queue...`
- **C3**：`JobKill(name="myjob", kill_cmd="echo would scancel 12345")` → kill_command 字段显示传入命令；kill_attempts append
- **C4**：launch "杀不死"任务（如 `bash -c 'trap "" TERM KILL; while true; do sleep 1; done'`），连续 3 次 `JobKill(name=X)` → 第 3 次返回里包含 L1 NOTE，meta 中 state=kill_failed
- **C5**：5 个独立的"杀不死"任务，每个 kill 3 次 → 第 5 个任务的第 3 次 JobKill 返回里同时包含 L1 NOTE + L2 WARNING（`Jobs(filter='stuck_kill')` 列出全部 5 个）

### D 组：JobArchive

- **D1**：B3 之后 `JobArchive(name="myjob")` → 返回 archived_at；本地 `<id>-meta.json` 移到 `archive/<id>-meta.json`；后续 `Jobs()` 不再列出；`Jobs(id=<id>)` 仍可查
- **D2**：state=running 的任务 `JobArchive(name="X")` → Error: `task 'X' state is 'running'...`
- **D3**：刚 launch 未观察过的任务（state="running" 是预置默认值）`JobArchive(name="X")` → 同 D2 的 Error（state="running" 触发拒绝）
- **D4**：归档后 `Bash(run_in_background=True, name="myjob", command="sleep 1")` → 允许（同名复用）；新任务 id 与归档任务不同；归档任务通过 `Jobs(id=<old_id>)` 仍可查
- **D5**：归档后远端 pid 文件 / log 文件仍存在（归档不删远端任何东西）
- **D6**：kill_failed 任务 `JobArchive(name="X")` 不带 as_zombie → Error: `cannot archive task 'X' in state 'kill_failed'...`
- **D7**：kill_failed 任务 `JobArchive(name="X", as_zombie=True)` → 接受，本地 `<id>-meta.json` 移到 `zombie/<id>-meta.json`（**不是 archive/**）；返回里包含 "Archived ... as ZOMBIE"；`Jobs(filter="zombies")` 列出该任务
- **D8**：5 个 zombie 任务的归档：第 5 次 `JobArchive(as_zombie=True)` 返回包含 escalation warning（实现按 `len(listdir(zombie/))` 计数）
- **D9**：stopped 任务 `JobArchive(name="X", as_zombie=True)` → Error: `as_zombie=True requires state=kill_failed...`
- **D10**：JobArchive 调用过程中**没有任何远端 SSH 调用**（可通过 packet capture 或 SSHConnection mock 验证）

### E 组：JobScript

- **E1**：`JobScript(name="myjob", script="echo hello", timeout=5)` → first-run 验证成功，返回 stdout=hello / exit_code=0；本地 `<id>-status.sh` 存在；远端 status.sh 缓存存在；meta `script_timeout=5`
- **E2**：E1 后 `Jobs(name="myjob")` → status_script_output 含 stdout=hello
- **E3**：`JobScript(name="myjob", script="sleep 30", timeout=2)` → Error: `status script first-run timed out after 2s...`; `Jobs(name="myjob")` 之后 status_script_output=null（本地脚本被清除，远端也被清除）
- **E4**：`JobScript(name="myjob", script="exit 2", timeout=5)` → 接受脚本，返回 first-run exit_code=2 + NOTE 提示
- **E5**：`JobScript(name="myjob", script="", timeout=5)` → 清除脚本：本地文件删除、meta script_timeout=null；远端缓存**保留**
- **E6**：`JobScript` 缺 timeout → schema validation 失败
- **E7**：手动删除远端 status.sh 缓存后 `Jobs(name="myjob")` → 自动从本地重新上传并执行成功

### F 组：sid 派生与跨 session 隔离

- **F1**：mcp server 启动后 `RemoteInfo` 返回里包含 `sid: <hex12> (source=ppid+starttime)`
- **F2**：手动 kill 然后 restart mcp server（父进程不变）→ 新进程 RemoteInfo 显示**相同**的 sid；之前的任务在 `Jobs()` 里仍可见
- **F3**：两个独立 mcp server 进程（模拟两个 Claude Code session）连同一 host → 各自看到自己的任务，互不可见（不同 sid 子目录）
- **F4**：同一 Claude Code session 连两个不同 host（两个 mcp 进程）→ 两个进程 sid 相同但 `<host>` 子目录不同，任务互不可见

### G 组：鲁棒性

- **G1**：`Bash(run_in_background=True, ...)` 在远端 mkdir log_path 父目录阶段网络断 → Error 含 "creating log_path parent for X" context；本地 `<id>-meta.json` 已清理
- **G2**：`Jobs()` 在远端 batched kill check 阶段超时 → 抛错中止，无 partial 结果
- **G3**：`Jobs(name=X)` 状态脚本运行卡死 → script_timeout 后返回，status_script_output.error 含 "timed out"，state / elapsed_sec 等字段正常
- **G4**：`JobKill` 在写本地 kill_requested_at 后、exec kill 前网络断 → kill_requested_at 已写；下次 `Jobs(name=X)` 刷新后 state=running/kill_failed + kill_requested_at 非 null（agent 可知 kill 未生效）
- **G5**：`Bash(run_in_background=True, ...)` 启动响应丢失但 pid 文件已写 → launch 自身同步 SFTP 兜底，取回 pid，任务正常入面板（覆盖在 A7）；如果 SFTP 兜底也失败 → 任务不入面板 + Error（覆盖在 A8）

### H 组：启动初始化

- **H1**：首次 mcp 启动 → `~/.local/share/remote-mcp/jobpane/<sid>/<host>/` 目录与 `archive/` / `zombie/` 子目录被创建；`next_id` 内容为 `"0"`
- **H2**：本地目录已存在 → 启动不重复创建（mkdir -p 幂等）；`next_id` 已存在则保留原值
- **H3**：本地目录所在路径不可写 → mcp 启动 fail-fast，stderr 打印明确 Error；进程退出码非 0

## 19. `exec_with_snapshot` 共享 helper

### 19.1 动机

v0.3.0 的面板工具大量需要"在远端跑一段 bash 命令"：

- `JobKill` 跑 `kill -TERM -- -<pid>` 或自定义 kill_cmd（packed with kill -0）
- `JobScript` 跑 status.sh first-run
- `Jobs` 单任务模式跑 status.sh、所有模式跑 `kill -0 + date` 批量
- `Bash` 前台路径本来就是"在远端跑命令"

这些场景共享需求：snapshot source 一致、`</dev/null` 一致、超时管理一致、结果收集一致。v0.2.x 这些逻辑只存在于 `tools/bash.py::_bash_foreground`，与 Bash 工具的"返回 string + 用户面向 Error wrap + cap 100KB"行为耦合。

### 19.2 抽取

放在 `connection.py`（与 `SSHConnection` 同文件，因为依赖 connection 对象）或新文件 `tools/_exec.py`：

```python
from dataclasses import dataclass

@dataclass
class ExecResult:
    stdout: str       # 已收到的 stdout（可能 partial 当 timed_out=True）
    stderr: str       # 已收到的 stderr（同上）
    exit_code: int    # paramiko 取得的 exit code；timed_out 时为 -1
    timed_out: bool   # True 表示触发 timeout（channel 已被强制 close）
    elapsed_sec: float

def exec_with_snapshot(conn, command: str, timeout: float) -> ExecResult:
    """
    Run command on remote with snapshot sourced + </dev/null.
    Wraps as: bash --noprofile --norc -c 'source <snapshot>; <command>' </dev/null

    Drain stdout/stderr separately; on timeout, channel.close() and return
    partial output with timed_out=True. Caller decides error formatting and
    output capping.

    Does NOT raise on remote errors (non-zero exit); does NOT raise on
    timeout (returns timed_out=True instead). Raises only on SSH-layer
    exceptions (paramiko.SSHException, OSError) — caller must handle these.
    """
    ...
```

### 19.3 调用方改造

- `tools/bash.py::_bash_foreground`：调 `exec_with_snapshot`，把 ExecResult 包装成 CC native 字符串（合并 stdout+stderr、附 `[Exit code: N]` 后缀、cap 100KB、Error 文本）
- `tools/job_kill.py`：调 `exec_with_snapshot` with 5s timeout；把 ExecResult 直接塞进返回的结构化字段
- `tools/job_script.py`：调 `exec_with_snapshot` with agent-provided timeout；ExecResult 直接塞进 `status_script_output` JSON
- `tools/jobs.py`：调 `exec_with_snapshot` with `op_timeout_default`；解析 stdout 拿到 state 列表

### 19.4 不在抽取范围内

后台 wrap 的 exec（§5.3.3）**不**走 `exec_with_snapshot`——因为它有特殊形态（外层是 setsid+nohup+&，需要短 timeout 等 BG_PID echo，解析返回是结构化字段而非通用输出）。这条路径保持在 `bash.py::_bash_background` 内独立处理。
