# 带宽与延迟

> English version: [bandwidth-and-latency.md](./bandwidth-and-latency.md)

网络约束不是一个脚注——它是驱动 remote-mcp 中几乎每一个非显而易见选择的设计强制函数。本文解释这一假设前提，追溯它如何塑造了每一项主要优化，并通过具体的数字让你感受到朴素方案的糟糕表现，从而真正理解优化方案的意义所在。

## 基线假设

remote-mcp 针对最苛刻的现实场景进行设计：蜂窝调制解调器或跨洲际 VPN 链路，持续吞吐量约为 100 KB/s 到 1 MB/s，RTT（往返时延）约为 200–1000 毫秒。

这些数字从两个独立维度产生影响：

**吞吐量**决定单位时间内可以传输多少数据。一个 500 KB 的源文件在这样的链路上需要 0.5–5 秒传输，而不是毫秒级。

**延迟（RTT）**决定每次来回交换需要多长时间，与数据量无关。即使是零字节的请求，最低成本也是一次 RTT。在 500 ms RTT 的情况下，三次顺序工具调用（每次都需要独立往返）在任何字节移动之前就已经产生了至少 1.5 秒的纯等待。在 200 ms RTT 的情况下，五次顺序的 Read 调用来探索一个模块，仅开销就至少 1 秒。

设计良好的 remote-mcp 会话应当同时最小化这两者：尽可能少传输字节，并将尽可能多的操作打包进单次往返。

## 优化一：Read 的远端 sed 切片

**没有它的问题：** 实现 Read 的朴素方式是「通过 SFTP 获取整个文件，然后在 Python 里切片」。对于小文件这是可以接受的。但对于一个 50 MB 的日志文件，而 agent 只想要第 8000–8020 行，朴素方案要传输 50 MB。在 100 KB/s 的速度下，这超过 8 分钟；即使在 1 MB/s 下也要 50 秒。

**优化方案：** Read 通过 exec 通道使用 `sed -n '{offset},{end}p; {end+1}q'`。`sed` 命令在远端主机上运行，仅提取所请求的行，只发回这些内容——无论文件总大小如何，通常只有几 KB。

**有无优化的代价对比：** 一个 50 MB 的文件，读取 20 行：50 MB vs. 约 2 KB。在 100 KB/s 下，8 分钟 vs. 不到 1 秒。

这个优化意味着 Read 的 `offset` 和 `limit` 参数不仅仅是为了方便——在高延迟链路上，它们是必不可少的。一个读取 100,000 行文件中前 2000 行的 agent 是在发出正确的信号；而一个为了搜索而读取整个文件的 agent 做的是错误的事情（应该使用 Grep）。

## 优化二：MultiRead——N 次往返压缩为一次

**没有它的问题：** 探索一个模块通常意味着依次读取几个相关文件：配置文件、模型定义、工具函数。每次 Read 都是一次独立的工具调用，也就是一次独立的往返。在 500 ms RTT 下，读取五个文件在任何内容到达之前就产生了 2.5 秒的纯延迟开销。

**优化方案：** MultiRead 接受一个 `{file_path, offset, limit}` 条目列表，构造一个对每个文件应用 `sed` 切片并用分隔符标记分隔结果的单一 shell 脚本，通过一次 exec 调用发送整个脚本，然后在客户端解析合并后的输出。五个文件，一次往返。

**有无优化的代价对比：** 在 500 ms RTT 下，读取 5 个文件：2500 ms 延迟开销 vs. 500 ms。内容传输时间在两种情况下是相同的。

MultiRead 是任何探索模式的正确默认选择。错误的模式是：读 file1，思考下一步读什么，读 file2。正确的模式是：决定需要哪些文件，一次全部读取。

## 优化三：FileStat——不传内容只获取元数据

**没有它的问题：** 在没有 FileStat 的情况下，检查「这个文件存在吗，有多大？」的自然方式要么是执行一个 Bash `stat` 命令，要么尝试 Read 后检查错误。Bash 需要一次往返。尝试 Read 需要一次往返*加上*如果文件存在就传输完整文件。对于一个 200 MB 的文件，「这个文件存在吗？」如果你试图读取它，就要付出 200 MB 的数据传输代价。

**优化方案：** FileStat 使用 SFTP 的 `stat()` 操作，通过单个 SFTP 消息返回结构化的元数据（大小、mtime、权限、类型）。响应以字节计，而非 KB 或 MB。它接受路径列表，所以你可以通过一次调用检查十个文件的存在性和大小。

**有无优化的代价对比：** 检查一个 200 MB 文件的元数据：200 MB + 1 RTT vs. 约 100 字节 + 1 RTT。FileStat 比 Bash 还要严格快，因为它复用了已存在的 SFTP 通道，而非打开新的 exec 通道。

## 优化四：MultiEdit——N 次编辑只传输一次文件

**没有它的问题：** 用三处独立修改重构一个文件需要三次 Edit 调用。每次 Edit 调用都会读取完整文件、做一处修改、再写回完整文件。三次编辑 = 六次完整文件传输。对于一个 50 KB 的源文件在 100 KB/s 速度下，这是三秒的传输时间，还不算 RTT。

**优化方案：** MultiEdit 接受一个编辑列表作为单次调用。它读取文件一次，顺序应用所有编辑（如果任何编辑失败则原子性地全部失败），写入一次。三次编辑 = 两次文件传输。

**有无优化的代价对比：** 一个 50 KB 文件，三次编辑：300 KB 总传输 vs. 100 KB 总传输。在 100 KB/s 下，3 秒 vs. 1 秒。随着编辑次数增加，比率进一步改善。

MultiEdit 还提供了原子性：如果第 2 个编辑失败（`old_string` 未找到，或找到多次），文件根本不会被写入。这比三次顺序 Edit（前两次成功，第三次失败，文件处于部分修改状态）的行为要好得多。

## 优化五：Grep 上下文标志——消除后续 Read

**没有它的问题：** 一个常见的工作流是：用 Grep 找到相关位置，然后用 Read 读取周围代码以了解上下文。这是两次工具调用、两次往返，以及 Read 的部分文件传输。

**优化方案：** Grep 支持 `-A`（after）、`-B`（before）和 `-C`（context）参数，可在 grep 输出本身中包含周围的行。「找到所有使用 `run_migration` 的地方并显示 5 行上下文」是一次带有 `context=5` 的单次 Grep 调用。grep 在远端主机上运行，在其输出中包含上下文行，并在一次响应中返回所有内容。

**有无优化的代价对比：** 查找 + 为上下文而 Read：2 次 RTT + 匹配区域的传输。带上下文的 Grep：1 次 RTT + 匹配区域的传输。在 500 ms RTT 下，每次搜索-检查循环可以稳定节省 500 ms。

## 优化六：Bash 后台模式——解除阻塞而不减少字节

**没有它的问题：** 在慢速网络上，一个 `pip install -r requirements.txt` 可能需要五分钟，一个 `npm run build` 可能需要三分钟。在此期间，Bash 工具处于阻塞状态，等待命令完成。agent 被阻塞，用户被阻塞，什么都做不了。

**优化方案：** `Bash(command="...", run_in_background=true)` 将命令包装在 `setsid nohup bash -c ... > /tmp/rmcp-bg-<uuid>.log 2>&1 </dev/null &` 中，启动它，捕获 PID，然后立即返回——通常只需几秒钟。agent 收到 PID、日志路径以及用于检查状态、读取输出、停止进程和强制停止进程的模板命令。

**有无优化的代价对比：** 这个优化不减少传输字节数。它消除的是挂钟阻塞时间。一个五分钟的构建仍然是五分钟的远端 CPU 工作——但 agent 在运行期间可以做其他事情，用户不必干盯着进度转圈。

日志通过 Read 使用 `offset=<last_line+1>` 增量读取，这意味着你可以轮询新输出而无需重传整个日志。结合 sed 切片优化，轮询一个增长中的构建日志是节省带宽的。

## 优化七：SSH 压缩

**优化方案：** SSH Transport 以 `compress=True` 运行，所有 SSH 流量在传输前经过 zlib 压缩。

源代码、配置文件和日志输出具有极高的可压缩性——ASCII 文本通常能达到 3–10 倍的压缩比。这意味着一个 100 KB 的 Python 文件可能以 15–30 KB 的压缩 SSH 负载传输。压缩在 paramiko 层透明发生，任何工具实现都无需感知它。

**代价：** 双端各需要少量 CPU 时间。在现代硬件上（包括 agent 侧的笔记本和任何合理的服务器），与网络节省相比，这个代价在实践中是无法测量的。

## 优化八：Write 的 SFTP mkdir

**没有它的问题：** V1 的 Write 工具在 SFTP 写入前先运行 `conn.exec("mkdir -p {parent}")`。这会打开一个新的 exec 通道、支付通道建立 RTT、运行命令、获取结果、关闭通道——然后再为写入本身打开 SFTP 会话。

**优化方案：** V2 使用 SFTP 自身的 `mkdir` 操作在已打开的 SFTP 通道上实现 `_sftp_mkdirs()`，无需新的 exec 通道。每级目录创建一条 SFTP 消息，而非 exec 通道开销加 SFTP 开销。

**有无优化的代价对比：** 每次向新目录写入时少一次通道开关周期。在 500 ms RTT 下，每次写入新路径节省 500 ms。

## 输出上限：防止意外的输出洪水

工具中有三种输出上限：

- Read 结果：上限 256 KB
- Bash 输出：上限 100 KB
- Glob 结果：上限 1000 条

这些上限不是通常意义上的带宽优化——它们是防止可能产生数 MB 甚至 GB 输出的命令的安全护栏。`find /` 会返回数百万行。`grep "e" /`（在所有地方搜索字母「e」）会返回 GB 级数据。意外发出此类命令的 agent 应该得到一个截断的结果并附有清晰的提示（`... [truncated to N bytes]`），而不是等待 30 分钟后得到一个内存溢出错误。

这些上限是针对单次调用的，因此产生合理输出的精准命令永远不会受到影响。

## 哪些代价是无法消除的

有些事情是无法优化掉的：

- Edit 始终需要读取和写入完整文件（需要验证 `old_string` 的唯一性）。100 KB 以下的源文件没问题；数 MB 的文件是已知的痛点。
- 任何工具调用的最低代价是一次 RTT。在 1000 ms RTT 的链路上，即使是一次简单的 FileStat 也有 1 秒的时间下限。
- 后台 bash 任务仍在远端主机上运行其完整时长；`run_in_background` 只是解除了 agent 的阻塞，并不会让计算更快。
