# 说明文档

> English version: [README.md](./README.md)

说明文档的作用是**阐明原委**。它提供背景脉络、讨论备选方案，并揭示设计背后的「为什么」。当你想要理解这个系统，而不仅仅是使用它时，请阅读这部分文档。

说明文档*不是*教程（不含操作步骤），也不是参考手册（不含详尽列表）。它是一场讨论。

## 可用的说明文档

| 主题 | 阅读理由 |
|-------|-------------|
| [架构概览](./architecture.md) | 建立心智模型——有哪些进程、使用什么协议、数据如何流转 |
| [设计决策](./design-decisions.md) | 理解关键选择（paramiko、stdio MCP、持久 bash、sentinel 协议……）以及被拒绝的备选方案 |
| [带宽与延迟](./bandwidth-and-latency.md) | 塑造了几乎所有非显而易见设计选择的约束——为何要服务端切片、为何要有 MultiRead/MultiEdit、为何 Bash 支持后台运行 |
| [多主机模型](./multi-host-model.md) | 为何每台主机拥有独立的服务器进程、为何不采用联邦模式、agent 同时操作 2-3 台主机时会发生什么 |
| [重连与 WARNING 协议](./reconnect-and-warning.md) | 为何我们不允许静默恢复连接断开、为何 WARNING 文本如此具体、哪些状态能保留、哪些不能 |
| [开发反馈循环](./feedback-loop.md) | 为何我们提供 `Feedback` 工具、它的用途是什么、它与开发周期有何关联 |

## 说明文档的写作规范（面向贡献者）

- 讨论，而非指令。写「我们选择 X 是因为 Y」，而非「如果你想要 Y，请执行 X」。
- 将概念相互关联（「持久 bash 会话使 sentinel 协议得以运作，而这反过来意味着……」）。
- 包含已考虑过的备选方案以及它们被拒绝的原因——否定空间同样具有参考价值。
- 通过交叉链接指向参考手册获取确切的事实；不要在这里重复那些内容。
- 允许表达观点和判断。说明文档是有立场的。
