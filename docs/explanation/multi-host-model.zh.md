# 多主机模型

> English version: [multi-host-model.md](./multi-host-model.md)

remote-mcp 的设计支持一种常见但并不简单的工作流：单个 Claude Code 会话同时操作两到三台远程主机。本文解释这种支持背后的设计选择、为何这个模型刻意保持简单，以及它在何处开始力不从心。

## 核心模型：每主机一个进程

每台远程主机由一个独立的 remote-mcp 操作系统进程代表。你需要独立注册各台主机：

```bash
claude mcp add --global remote-prod -- python -m remote_mcp --host prod
claude mcp add --global remote-gpu  -- python -m remote_mcp --host gpu
```

Claude Code 每台主机看到十个工具，以注册名称为命名空间：

```
mcp__remote-prod__Read
mcp__remote-prod__Bash
mcp__remote-prod__Grep
...
mcp__remote-gpu__Read
mcp__remote-gpu__Bash
...
```

这是最简单的可行模型。每个进程与其他进程完全隔离——独立的内存、独立的 SSH 连接、独立的 bash 会话。它们在用户文件系统以下的任何层面都不会相互干扰（两者都可能读/写同一个 NFS 挂载，但那是用户的问题，不是我们的问题）。

## 为何不采用联邦设计？

显而易见的备选方案是一个单一的 remote-mcp 进程管理所有已配置的主机，暴露 `Read(host="prod", path="...")` 这样的工具。这样可以节省每台主机一个进程的开销，并允许共享连接池。

我们拒绝了这种方案，原因如下：

**Schema 复杂性。** 每次工具调用都带一个 `host` 参数是一个新的错误来源。agent 可能忘记它，或者传入错误的主机名，或者没有意识到 `host` 是必须的。每主机命名空间使主机标识成为结构性的——它不可能被省略。

**爆炸半径。** 一台主机的连接管理代码中的 bug 可能会破坏联邦进程中所有主机的状态。一台主机的进程崩溃不会影响其他主机。

**实现简单性。** 联邦模型需要路由逻辑、单进程内的每主机状态管理，以及对目标不同主机的并发工具调用的仔细处理。每进程模型没有这些。每个进程都是单租户服务器。

**用例不要求这样做。** 这套设计明确针对 2-3 台主机，而非一个机群。两个额外的操作系统进程和两条额外的 SSH 连接，对于开发者的机器来说不是一个有意义的资源成本。如果你在管理十台主机，你可能应该使用合适的配置管理系统，而不是对话式 AI agent。

## `[host=X cwd=Y]` 的作用

每次 Bash 工具的结果都以如下一行开头：

```
[host=prod cwd=/home/ubuntu/myproject]
```

这专门为多主机会话而存在。当 agent 在主机之间切换——在 `prod` 上运行迁移、在 `gpu` 上检查模型、在 `db` 上查询数据库——工具结果按顺序出现在对话上下文中。没有主机前缀，agent 就无法可靠地判断哪台主机的 bash 会话处于哪个工作目录。有了前缀，每条结果都是自我说明的。

cwd 从 sentinel 协议中捕获：每条命令都向 bash stdin 追加 `echo "RMCP_SENTINEL_{uuid}_EXIT_$?_CWD_$(pwd)"`，bash 会话从 sentinel 行中解析当前工作目录。这发生在协议层面，不是一条额外的命令，因此没有额外的 RTT 成本。

## WARNING 消息中主机名称的作用

当 SSH 连接断开并重连时，工具结果以如下内容为前缀：

```
[WARNING] SSH connection to prod was lost and has been re-established.
```

WARNING 中的主机名称与 Bash 结果中的主机前缀具有相同的作用：在多主机会话中，agent 需要知道哪台主机丢失了其状态，而不仅仅是某台主机丢失了。一个模糊的「连接已重建」消息会让 agent 不清楚哪个 bash 会话被重置了、哪些工作流需要重新启动。关于完整讨论，请参阅[重连与 WARNING 协议](./reconnect-and-warning.md)。

## 使用多台主机时 agent 应知道的

**尽可能将工作保持在单台主机内。** 一次操作跨越的主机越多，协调开销就越高。一个每隔几次工具调用就在 `prod` 和 `gpu` 之间切换的 agent，在可以用一次 RTT 的地方付出了两次 RTT 的代价。

**跨主机文件传输走 Bash，而非 Read+Write。** agent 不应该通过 `mcp__remote-prod__Read` 读取文件然后通过 `mcp__remote-gpu__Write` 写入来复制文件。这会双重传输文件：从 `prod` 到本地机器，再从本地机器到 `gpu`。正确的方式是在能访问另一台主机的那台主机上执行 `Bash("scp prod:/path/to/file /destination")`——这要求用户在主机之间配置好 SSH 密钥。

**状态是主机本地的。** 在 `prod` 的 bash 会话上执行 `cd` 对 `gpu` 的 bash 会话没有任何影响。在一台主机上 export 的环境变量不会出现在另一台主机上。这是显而易见的，但值得明确说出：agent 不能假设在一台主机上完成的设置会延续到另一台主机。

## 这个模型的局限性

每主机进程模型线性扩展。每增加一台主机会增加：
- 一个操作系统进程
- 一条 SSH TCP 连接
- 一个持久 bash 通道
- 一个 SFTP 客户端

2-3 台主机是舒适的。在一台拥有 16 GB 内存的笔记本上运行 10 台主机，你面对的是 10 个 Python 解释器实例（每个基线约 30-50 MB）、10 条 SSH 连接（小）和远端的 10 个 bash 进程。这大约是 300-500 MB 的本地内存和远端 10 个 SSH 会话——可能没问题，但开始变得有些不舒服了。

超过 10 台主机，正确的答案是不同的架构：单一的连接池守护进程、共享的 SFTP 会话，或者一个合适的远端执行框架。这被明确列为未来工作。目前，如果你同时处理多台主机，请考虑你是否真的需要它们全部同时激活，或者串行工作流（注册、工作、注销）是否足够。

## 非目标：跨主机原语

没有 `Copy(from_host, to_host)` 工具，也没有 `MultiHostBash(hosts=[...], command=...)`。这些被考虑过，并被明确推迟。

跨主机操作真的很难做对——错误处理、部分成功、回滚和原子性在涉及多条 SSH 连接时都会变得戏剧性地更加复杂。Bash + scp 的变通方案处理了实践中实际出现的情况，代价是要求用户在主机之间配置 SSH 信任。这对 2-3 台主机的用例来说是可接受的权衡。
