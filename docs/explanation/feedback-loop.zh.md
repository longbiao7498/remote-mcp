# 开发反馈循环

> English version: [feedback-loop.md](./feedback-loop.md)

remote-mcp 附带了一个 `Feedback` 工具。它不寻常：大多数工具的存在是为了在远端主机上做某件事，而 Feedback 的存在是为了让 agent 能够与维护者对话。本文解释这个工具为何存在、它捕获什么，以及它如何与开发周期相连接。

## 它解决的问题

像 remote-mcp 这样的工具的开发者面临一个根本性的可见性问题。工具的使用者主要是 AI agent，而非人类。当 agent 遇到 bug——schema 不匹配、输出混乱、意外的超时——它会自我适应并继续。它可能用不同的参数重试、切换到不同的工具，或者只是向用户报告一个模糊的失败。维护者永远不会听说。

当 agent 意识到「如果这个工具有一个 context 参数，这个工作流会简单得多」时，这个想法无处可去。agent 完成任务，观察就这样消散了。

传统的 bug 报告需要在失败的时刻有人工介入。如果用户注意到问题并愿意提交 issue，维护者会收到反馈。如果没有，维护者就不会知道。对于一个主要在 agent 层运作、低于用户感知阈值的工具来说，这个反馈率接近于零。

Feedback 工具是一个低摩擦的逃生阀：agent 可以当场记录它观察到的内容，不打断自己的任务，而维护者可以稍后查阅这些观察。

## 它捕获的内容

Feedback 记录两类内容：

**`bug`**：工具的行为方式与预期不符。schema 与 Claude Code 的原生工具不一致。错误消息的措辞与原生等价工具不同。输出被意外截断。一个本应成功的命令超时了。这些是维护者需要知道以修复的内容。

**`enhancement`**：agent 遇到了需要比必要更多往返的工作流，或者希望某个工具有一个它没有的参数，或者意识到一个新工具可以消除两到三个现有工具调用的某种模式。这些是维护者需要知道以改进的内容。

每条记录包含：时间戳、主机名称、会话 PID、类别、一行摘要，以及可选的详细信息。主机名称和会话 PID 使维护者在需要深入调查时能够将反馈条目与服务器日志关联起来。

关于精确的参数文档，请参阅 [Feedback 工具参考手册](../reference/tools/feedback.md)。

## 为何仅限本地，没有遥测

`details` 字段可以包含代码片段。一个解释 Glob bug 的 agent 可能会包含它使用的实际文件路径和模式。一个建议 MultiWrite 增强的 agent 可能会包含它正在处理的项目中的示例代码。那段代码属于用户的项目，在未经明确同意的情况下，它不应该离开用户的机器。

Feedback 工具写入本地 JSONL 文件（默认为 `~/.local/share/remote-mcp/feedback.jsonl`），仅此而已。没有网络请求，没有分析端点，没有后台同步。用户完全拥有这个文件。如果他们想与维护者分享反馈，可以发送文件或粘贴相关条目——这个决定由他们做出。

这不是未来版本中要解决的限制，而是一个蓄意为之的隐私立场。

## 开发反馈循环如何运作

预期的循环是：

1. agent 将 remote-mcp 用于真实项目的真实工作。
2. 当它注意到 bug 或有改进想法时，调用 `Feedback` 然后继续工作。
3. 维护者定期读取 `~/.local/share/remote-mcp/feedback.jsonl`——在一次会话结束后、在一周结束时，或在处理问题时。
4. 维护者去重条目，识别模式，并根据 agent 在实践中实际遇到的内容来确定修复或新功能的优先级。
5. 修改在下一个版本中发布，循环重复。

这与用户提交 GitHub issue 有根本性的不同，因为观察发生在 agent 时间（当 agent 正处于工作流中、上下文最丰富的时候），而非用户时间（数小时后，用户模糊地记得有什么不对劲）。agent 捕获了精确的工具调用、精确的输出和精确的期望——这些细节几乎总是在人类提交 bug 报告之前就丢失了。

## 它不适用于什么

Feedback 不是一个通用的笔记工具。它不用于记录关于用户代码库、远端系统健康状况或任何与 remote-mcp 工具本身无关的观察。MCP schema 中的工具描述对此是明确的：「记录关于 remote-mcp 工具本身的 bug 或改进想法（不是关于用户的代码或远端系统）。」

一个为远端主机的磁盘已满错误或用户 Python 代码中的语法错误提交 Feedback 条目的 agent，是在误用这个工具。这些内容属于用户自己的追踪系统，而不是 remote-mcp 的反馈文件。

## 为何是专用工具而非使用 Write

三个原因：schema 一致性、原子性和引导性。

**Schema 一致性。** Feedback 条目有一个定义好的结构：时间戳、主机、类别、摘要、详细信息、session_pid。如果 agent 直接写文件，它必须在每次调用时都记住这个 schema。Feedback 工具强制执行它，并正确填充自动字段（时间戳、主机、PID），无需依赖 agent 的记忆。

**原子性。** 多个 remote-mcp 进程（每台主机一个）共享同一个反馈文件。如果两个 agent 在两台主机上同时调用 Feedback，原始的文件写入可能会交错它们的 JSON。POSIX 保证在 `PIPE_BUF` 字节（通常为 4 KB）以下的 `write()` 调用对普通文件是原子的——而单条 Feedback 条目远低于该限制。工具依赖这个保证；shell 的 `echo >> file` 方式不会提供这个保证。

**引导性。** 一个有着写得很好的描述、在工具列表中可见的专用工具，会提醒 agent 这个通道的存在。写文件的方式会要求 agent 从上下文中记住文件路径、schema 和用途——全部靠自己。Feedback 工具是自我说明的，而 Write 不是。
